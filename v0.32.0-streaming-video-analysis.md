## 1. Introduction

Traditional video analysis processes the entire video file before producing any results. This creates a fundamental latency wall: for a 2-hour video at ~3-4 minutes processing time, the user waits minutes before seeing any output.

**Streaming video understanding** is an emerging paradigm where models and pipelines process video frames incrementally as they arrive, producing results with minimal latency. This is critical for:

- **Live monitoring** (surveillance, OBS recordings, RTSP camera feeds)
- **Interactive Q&A** (ask questions while video is still being recorded)
- **Long-form content** (podcasts, lectures — chunked processing reduces time-to-first-answer)
- **Resource-constrained environments** (RTX 4070 with 12GB VRAM must manage memory across chunks)

The landscape has exploded in 2025-2026, with dedicated streaming architectures appearing at ICLR 2026, ECCV 2026, CVPR 2025/2026, and NeurIPS 2025/2026. This document surveys the key approaches and distills design principles into our v0.32.0 Streaming Pipeline.

## 2. StreamingVLM (ICLR 2026)

**Paper:** [arXiv:2510.09608](https://arxiv.org/abs/2510.09608) — "StreamingVLM: Real-Time Understanding for Infinite Video Streams"  
**Authors:** Ruyi Xu, Guangxuan Xiao, Yukang Chen, Liuning He, Yao Lu, Song Han  
**Venue:** ICLR 2026  
**Code:** GitHub (linked in paper)

### Core Innovation

StreamingVLM solves the fundamental problem of near-infinite video streams: processing entire videos with full attention leads to quadratic costs, while simple sliding windows break coherence or suffer from redundant recomputation.

**Three-component KV cache management:**

1. **Attention sinks** — Reuse cached states from initial tokens (known phenomenon from StreamingLLM)
2. **Short window of recent vision tokens** — Maintains a rolling buffer of ~512 recent visual tokens
3. **Long window of recent text tokens** — Preserves up to ~4096 recent text tokens for coherent generation

### Training Strategy

A simple SFT strategy applies full attention on **short, overlapped video chunks** (typically 8-16 seconds with 2-4 second overlap). This mimics the inference-time attention pattern without training on prohibitively long contexts.

Key insight: **Training on short overlapping chunks transfers to streaming inference** — the model learns to handle the KV cache pattern without ever seeing million-token sequences during training.

### Inf-Streams-Eval Benchmark

New benchmark with videos averaging over **2 hours** that requires dense, per-second alignment between frames and text.

**Results:**
- 66.18% win rate against GPT-4O mini
- Stable real-time performance at **up to 8 FPS on a single H100**
- +4.30 on LongVideoBench
- +5.96 on OVOBench Realtime

### Relevance to v0.32.0

While StreamingVLM is a model-level solution (KV cache management inside the VLM), the **chunked processing with overlap** pattern directly inspired our streaming pipeline architecture:

| StreamingVLM Concept | v0.32.0 Analog |
|---------------------|----------------|
| Short overlapped video chunks (8-16s) | Configurable chunk_duration (default 30s) |
| Overlap for context continuity | overlap parameter (default 2s) |
| Incremental KV cache | Incremental ChromaDB indexing |
| No full-video waiting | Generator-based yield of StreamingChunkResult |

## 3. ThinkStream (ECCV 2026)

**Paper:** "Thinking in Streaming Video" — linked via [Awesome Streaming Video Understanding](https://github.com/sotayang/Awesome-Streaming-Video-Understanding)  
**Venue:** ECCV 2026  
**Approach:** Watch–Think–Speak streaming reasoning

### Core Innovation

ThinkStream introduces a **Watch-Think-Speak** framework for streaming video reasoning:

1. **Watch** — Continuously ingests video frames, updating a compressed streaming memory (RCSM)
2. **Think** — Uses the compressed memory to reason about when to speak (proactive decision)
3. **Speak** — Generates responses only when needed, using the compressed reasoning history

**Key technique:** Reasoning Compressed Streaming Memory (RCSM) replaces outdated visual tokens with compressed reasoning history, achieving low latency and memory efficiency.

### Relevance to v0.32.0

The Watch-Think-Speak pattern maps to our pipeline architecture:

| ThinkStream | v0.32.0 |
|-------------|---------|
| Watch: Frame ingestion | FFmpeg segment extraction |
| Think: Compressed memory | Accumulated scenes + transcripts across chunks |
| Speak: Response generation | Incremental indexing → immediate Q&A availability |

## 4. Streamo: Streaming Video Instruction Tuning

**Paper:** "Streaming Video Instruction Tuning" (December 2025, arXiv)  
**Approach:** State-Token Unified Triggering

### Core Innovation

Streamo introduces explicit response state tokens:

- **Silence** — Model predicts no response needed
- **Standby** — Model is listening/processing
- **Response** — Model generates a reply

These are integrated into the autoregressive sequence, so deciding **when** to respond and **what** to say becomes a single unified problem. A focal-weighted loss mitigates the extreme class imbalance (most frames = Silence).

### Relevance to v0.32.0

The state-token concept inspires our **live mode** design: the pipeline is always "Standby" (polling), transitions to "Response" when enough new content is available (chunk_duration reached), then back to "Standby."

## 5. MMDuet2 (ICLR 2026)

**Paper:** "MMDuet2: Enhancing Proactive Interaction of Video MLLMs with Multi-Turn Reinforcement Learning"  
**Venue:** ICLR 2026

### Core Innovation

Formulates proactive interaction as a **per-turn text decision** where the model outputs either a response or "NO REPLY." Trained via multi-turn RL with a PAUC-inspired reward that encourages early and correct responses without requiring reply-time annotations.

### Relevance to v0.32.0

The proactive decision-making fits our live-mode design pattern: the pipeline should decide **when** to process a chunk based on availability of new content, not just on fixed timers.

## 6. StreamingClaw

**Paper:** StreamingClaw Technical Report (March 2026, arXiv)  
**Approach:** Dedicated Trigger Tokens + Unified Agent Framework

### Core Innovation

Customizes dedicated trigger tokens for different scenarios (question-answering, commentary, anomaly detection). Proposes a unified agent framework that addresses the fundamental limitations in real-time perception-action closed loops.

**Key insight:** Different streaming scenarios need different trigger strategies — a unified agent can dispatch to the appropriate trigger based on context.

## 7. LION-FS: Fast & Slow Video-Language Thinker

**Paper:** "LION-FS: Fast & Slow Video-Language Thinker as Online Video Assistant"  
**Venue:** CVPR 2025

### Core Innovation

Two-path architecture:

1. **Fast Path** — Efficiently determines **when** to respond (via EOS token prediction)
2. **Slow Path** — Uses multi-granularity keyframe augmentation to generate detailed responses only when needed

EOS-based triggering: model predicts [EOS] token to stay silent or generates text to respond at each frame.

### Relevance to v0.32.0

The Fast-Slow dichotomy maps to our design:
- **Fast path (chunked file mode)**: Quick FFmpeg segment extraction + pipeline processing, yielding results immediately
- **Slow path (full video mode)**: Full pipeline with all stages for final merged index

Other streaming models using EOS-based triggering:
- **LiveCC** (CVPR 2025) — Streaming speech transcription at scale
- **VideoLLM-MoD** (NeurIPS 2024) — Mixture-of-Depths vision computation
- **VideoLLM-EyeWO** (NeurIPS 2025) — Active perception, requests high-res frames when uncertain
- **ProAssist** (EMNLP 2025) — Proactive assistant dialogue from streaming egocentric video
- **AssistPDA** (2025) — Online video surveillance, anomaly prediction via EOS probability

## 8. Awesome-Streaming-Video-Understanding Survey

**Repository:** [github.com/sotayang/Awesome-Streaming-Video-Understanding](https://github.com/sotayang/Awesome-Streaming-Video-Understanding)  
**Survey PDF:** Included in the repo — "Towards Online Interactors: A Comprehensive Survey on Streaming Video Understanding"

### Key Categories from the Survey

The survey organizes streaming video understanding into two fundamental challenges:

**Proactive Decision-Making (When to Act):**
- Token-Driven Triggering via EOS / Action Token (LiveCC, VideoLLM-MoD, ProAssist, AssistPDA, Streamo, MMDuet2)
- Dedicated Classification Heads / Detectors (StreamingClaw)
- Uncertainty & Perplexity Validation
- Visual Change / Event-based Trigger

**Efficient Resource Management (How to Sustain):**
- KV Cache Management & Eviction (StreamingVLM)
- Hierarchical Memory & Summarization (ThinkStream's RCSM)
- Retrieval-Augmented Mechanisms
- Computational Efficiency & Sparse Computing (VideoLLM-MoD)

### v0.32.0 Categorization

Our pipeline falls under **Efficient Resource Management — Retrieval-Augmented Mechanisms** at the infrastructure level, with the **Watch-Think-Speak** proactive pattern from ThinkStream informing the live mode.

## 9. Architecture Comparison & Synthesis

### Model-Level vs Pipeline-Level Streaming

| Approach | Level | Requirements | RTX 4070 Fit |
|----------|-------|-------------|--------------|
| StreamingVLM | Model (KV cache) | Custom VLM fine-tuning, H100 | No (needs H100 FP8) |
| ThinkStream | Model (RCSM) | Custom training | Unknown |
| Streamo | Model (state tokens) | SFT on instruction data | Possibly |
| **v0.32.0 Pipeline** | **Pipeline (chunked)** | **FFmpeg + existing models** | **✅ Yes (12GB)** |

### Tradeoff Analysis

| Metric | Chunked (v0.32.0) | Full-Video (existing) | Streaming Model |
|--------|-------------------|----------------------|----------------|
| Latency to first result | ✅ Chunks processed incrementally | ❌ Wait for full video | ✅ Immediate (model-level) |
| Total processing time | ⚠️ Slightly higher (segment overhead) | ✅ Single pass | ✅ Single pass |
| Memory efficiency | ✅ Models unloaded per-chunk | ⚠️ All models loaded sequentially | ❌ KV cache grows |
| Complexity | ✅ Uses existing pipeline | ✅ Already built | ❌ New model training |
| Live/streaming support | ✅ Built-in (file_watch mode) | ❌ Not supported | ✅ Native |
| Cross-chunk context | ⚠️ Overlap + merging | ✅ Full context | ✅ KV cache carries over |

### Key Insight

For our **RTX 4070 (12GB VRAM)** environment, the **chunked pipeline approach** is the optimal choice:

1. No new model fine-tuning required
2. Existing GPU models already fit per-chunk (they were designed for full-video processing)
3. Chunked processing naturally enables live/streaming workflows
4. Incremental indexing makes content searchable immediately
5. Pipeline cleanup between chunks keeps VRAM usage predictable

## 10. v0.32.0 Design Decisions

### Decision 1: FFmpeg Stream Copy for Segment Extraction

**Chosen:** `ffmpeg -c copy` (stream copy, no re-encoding)  
**Alternative:** Re-encode to keyframe-aligned segments

**Rationale:** Stream copy is ~100x faster than re-encoding. The tradeoff is that segment boundaries may not align exactly with keyframes, causing a few seconds of decoder delay on each segment. This is acceptable for our analysis pipeline (we extract keyframes with `-ss` seek anyway).

### Decision 2: 30-Second Default Chunk Duration

**Chosen:** 30 seconds  
**Rationale:** Balances latency to first result (30s of video → ~3-5s processing) against pipeline overhead. Matches the fixed_60s and sliding_30s chunking strategy already used in the RAG module.

### Decision 3: Generator-Based API

**Chosen:** `Generator[StreamingChunkResult, None, VideoIndex]`  
**Rationale:** Allows callers to `for result in pipeline.process_streaming(...)` and get results incrementally while the pipeline continues processing. Compatible with both synchronous CLI usage and async wrapper patterns.

### Decision 4: Incremental vs Batch Indexing

**Chosen:** Configurable via `incremental_index` parameter (default: True)  
**Rationale:** Incremental indexing makes content searchable immediately but creates multiple ChromaDB entries per video. Batch indexing creates a single index entry but requires waiting for the full video. The user can choose based on their workflow.

### Decision 5: Pipeline Cleanup Between Chunks

**Chosen:** Call `pipeline.cleanup()` after each chunk  
**Rationale:** With 12GB VRAM, loading all models for a full video and then unloading is the standard pattern. Between chunks, unloading ensures we don't accumulate GPU memory across chunks. The overhead of model (re)loading is ~2-5s per chunk, acceptable for the streaming use case.

### Decision 6: Overlap for Context Continuity

**Chosen:** 2-second overlap (configurable)  
**Rationale:** Short enough to minimize redundant processing, long enough to ensure scene boundaries near chunk edges are captured in full in at least one chunk.
