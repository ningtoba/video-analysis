## 1. Executive Summary

The field of video MLLM fine-tuning has matured dramatically in 2025–2026. Key developments:

| Area | State of the Art | RTX 4070 Feasible? |
|------|------------------|---------------------|
| **Adapter-based SFT** | LoRA/DoRA on 2B–8B models | ✅ Yes (2B comfortably, 8B with QLoRA/gradient checkpointing) |
| **Preference Optimization** | DPO, GRPO, ST-GRPO, J-GRPO | ⚠️ DPO yes; GRPO variants need careful memory management |
| **Pre-training / Continual** | Video-text alignment pre-training | ❌ Requires multi-GPU or large datasets |
| **Reinforcement Learning** | GRPO-based video reasoning | ⚠️ Possible on 2B models with LoRA |
| **Datasets** | Streamo-Instruct-465K, Live-CC-5M, CoTasks | ✅ Available on HuggingFace |

**Best paths for self-hosted RTX 4070:**
1. **LoRA SFT** on SmolVLM2-500M or SmolVLM2-2.2B for domain-specific video captioning
2. **QLoRA** on Qwen2.5-VL-3B or Qwen3-VL-30B-A3B (3B active) for video QA
3. **DPO** on VideoChat-Flash-2B for preference alignment (reduce hallucination)
4. **Offline GRPO** on 2B-class models for reasoning improvement

## 2. Adapter-Based Approaches

### 2.1 LoRA (Low-Rank Adaptation)

The dominant paradigm for video MLLM fine-tuning. Virtually every paper uses LoRA.

**Key Papers:**
- **APT-Tune** (arXiv:2606.18586, June 2026) — Parameter-efficient recipe using **only 11M LoRA parameters** on Qwen3-VL-2B for causal physical video understanding. Demonstrates standard rank=32, alpha=64 config on the LLM backbone only (not vision encoder).
- **Hi-GaTA** (arXiv:2605.11208, May 2026) — Surgical video report generation using LoRA on LLM backbone while freezing temporal adapter and vision encoder.
- **VTG Paradigm Study** (arXiv:2604.08966, CVPR 2026 Workshop) — Controlled comparison of VTG output paradigms using LoRA fine-tuning on **SmolVLM2, FastVLM, and Molmo2** with consistent protocols. Provides reference LoRA configs and training throughput baselines.

**Recommended LoRA Config (RTX 4070):**
```
r=16, alpha=32, target_modules=["q_proj", "v_proj", "k_proj", "o_proj"]
Dropout: 0.05
Freeze: vision encoder, temporal adapter
Trainable params: 0.1–1% of total model
```

### 2.2 QLoRA (Quantized LoRA)

Enables fine-tuning of larger models on 12GB VRAM via 4-bit NF4 quantization.

**Status:** QLoRA is widely used in the open-source ecosystem (LLaMA-Factory, Unsloth) but no dedicated video MLLM QLoRA paper was found. However, it's trivially applicable:

- **SmolVLM2-2.2B** can be fine-tuned at BF16 with standard LoRA (~8GB VRAM)
- **InternVideo3-8B** requires QLoRA (4-bit) = ~6-7GB VRAM, leaving 5GB for data/training
- **Qwen2.5-VL-3B** or **Qwen3-VL-30B-A3B (3B active)** both fit with QLoRA

**Reference implementation:** Unsloth (https://github.com/unslothai/unsloth) supports all Qwen2.5-VL variants with QLoRA.

### 2.3 DoRA (Weight-Decomposed Low-Rank Adaptation)

DoRA decomposes weights into magnitude and direction components for finer-grained adaptation.

**Status:** No dedicated video MLLM DoRA papers published. DoRA is available in LLaMA-Factory and PEFT library. Apply directly to any video MLLM backbone. Slightly higher VRAM cost than LoRA (~5-10%) with marginal accuracy gains reported in text-domain papers. **DoRA is worth experimenting with on SmolVLM2-500M** (lowest risk due to small model size).

### 2.4 Temporal Adapters (Non-LoRA Parameter-Efficient Methods)

Several papers propose custom temporal adapters that are more architectural than fine-tuning methods, but highly relevant:

- **Hi-GaTA** (arXiv:2605.11208) — Hierarchical Gated Temporal Aggregation Adapter. Temporal pyramid with text-conditioned dual cross-attention + cross-level gated fusion. Lightweight; frozen during LLM fine-tuning.
- **M²LA** (InternVideo3, arXiv:2606.12195) — Multi-Modal Memory-Latency Adapter for KV-cache compression. Not a fine-tuning technique per se, but enables longer video context during training.
- **Continuous Temporal Decoding** (arXiv:2604.08966) — Output paradigm that achieves best efficiency-accuracy trade-off for temporal grounding. Works as a lightweight head adaptor.

**Recommendation:** Don't reimplement temporal adapters. Use existing models' built-in mechanisms (InternVideo3's Perceiver resampler, VideoChat-Flash's hierarchical compression).

## 3. Supervised Fine-Tuning (SFT) for Video

### 3.1 Video Captioning SFT

| Paper | Model | Dataset | Key Insight |
|-------|-------|---------|-------------|
| **Streamo** (CVPR 2026) | Streaming video LLM | Streamo-Instruct-465K | Real-time narration + event captioning + temporal grounding in a unified SFT pipeline |
| **VidEmo** (NeurIPS 2025) | Video emotion FM | Emo-CFG (2.1M samples) | Two-stage: curriculum emotion learning then affective-tree RL |
| **ViCA2** (2025) | ViCA2-7B | ViCA-322K | Dual vision encoder (SigLIP + Hiera) + spatial QA data for visuospatial reasoning |
| **CoTasks** (2025) | LLaVA-Video / Qwen2.5-VL | CoTasks-enhanced NeXT-QA | Decomposes complex QA into frame localization → entity tracking → relation extraction |
| **LiveCC** (2025) | Video LLM | Live-CC-5M + Live-WhisperX-526K | Uses ASR transcripts as cheap supervision for video-text alignment at scale |

### 3.2 Video QA SFT

The standard SFT pipeline for video QA follows a **three-stage** procedure:

1. **Stage 1 — Vision-Language Alignment Pre-training:** Large-scale image/video-text pairs (e.g., Live-CC-5M, VideoCC-3M). Freeze LLM, train projection layer or Q-Former. **Not feasible on RTX 4070** without significant data subsampling.
2. **Stage 2 — Multi-task Instruction Tuning:** Mix of captioning, QA, reasoning tasks. All parameters or LoRA. **Feasible on RTX 4070** with LoRA.
3. **Stage 3 — Preference Alignment:** DPO/GRPO on video QA pairs. **Feasible on RTX 4070** with LoRA.

**Practical SFT Recipe for RTX 4070:**

```python
# Using transformers.Trainer with PEFT
from peft import LoraConfig, get_peft_model
from transformers import AutoModelForCausalLM, AutoProcessor, TrainingArguments, Trainer

# Model: SmolVLM2-500M, SmolVLM2-2.2B, or Qwen2.5-VL-3B
model = AutoModelForCausalLM.from_pretrained(
    "HuggingFaceTB/SmolVLM2-500M-Video-Instruct",
    torch_dtype=torch.bfloat16,
    device_map="cuda",
)

peft_config = LoraConfig(
    r=16,
    lora_alpha=32,
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
)
model = get_peft_model(model, peft_config)

training_args = TrainingArguments(
    output_dir="./video-lora-output",
    per_device_train_batch_size=1,  # Crucial for 12GB
    gradient_accumulation_steps=8,
    learning_rate=2e-4,
    warmup_steps=100,
    num_train_epochs=3,
    logging_steps=10,
    save_steps=500,
    fp16=True,  # or bf16=True if supported
    gradient_checkpointing=True,
    dataloader_pin_memory=False,
    remove_unused_columns=False,
)
```

**Batch size guidance for RTX 4070 12GB (LoRA, BF16, gradient checkpointing):**

| Model | Batch Size | Grad Accum | Effective Batch |
|-------|-----------|------------|-----------------|
| SmolVLM2-500M | 4 | 4 | 16 |
| SmolVLM2-2.2B | 1 | 8 | 8 |
| VideoChat-Flash-2B | 1 | 8 | 8 |
| Qwen2.5-VL-3B | 1 | 8 | 8 |
| Qwen3-VL-30B-A3B (3B active, QLoRA) | 1 | 8 | 8 |
| InternVideo3-8B (QLoRA 4-bit) | 1 | 8 | 8 |

## 4. Preference Optimization

### 4.1 DPO (Direct Preference Optimization)

**Applicability to Video MLLMs:** Well-established. Multiple papers confirm DPO reduces hallucination and improves video QA quality.

**Key Papers:**
- **UnifiedReward** (arXiv:2503.05236, March 2025/updated Feb 2026) — Unified reward model for multimodal understanding & generation. Uses DPO on both image and video tasks. Large-scale human preference dataset. Shows that joint image+video reward training yields mutual benefits. **The most comprehensive reference for video DPO.**
- **DPO for Video LMMs from LM Reward** (arXiv:2404.01258, April 2024) — Pioneering work. Uses detailed video captions as proxy to enable language-only reward models for scoring video QA pairs, then applies DPO.
- **VIDR (Video Instruction Diversity Reward)** — Not a paper, but a technique seen in open-source. Uses intra-sample diversity as reward signal.

**DPO Training Setup (RTX 4070):**
```
Reuse the same LoRA config from SFT. Start from the SFT checkpoint.
DPO β (temperature): 0.1
Data format: (video_frames, chosen_answer, rejected_answer)
Loss: DPO pairwise loss applied to the LLM head
Memory: Pre-compute log probabilities of chosen/rejected for efficiency
```

**Estimated VRAM:** DPO adds ~1-2GB over SFT (need to store log-probs of both responses). On 2B models: ~8GB total. On 8B QLoRA: ~9-10GB total — tight but feasible.

### 4.2 GRPO (Group Relative Policy Optimization)

GRPO has become the dominant RL method for video MLLMs in 2026, particularly for reasoning tasks.

**Key Papers:**
- **CARE** (arXiv:2606.19927, June 2026) — Competence-Aware Reward Shaping for adaptive reasoning length in video MLLMs. Integrated into GRPO pipeline. Shows **inverted-U reasoning length trajectory** — models naturally learn to reason more efficiently. Open source at https://github.com/1Pansy/Video-CARE.
- **Insight-V++** (arXiv:2603.18118, March 2026) — Introduces **ST-GRPO** (Spatial-Temporal GRPO) and **J-GRPO** (Joint GRPO). Two-agent architecture (reasoning agent + summary agent) with iterative self-improvement. Tested on LLaVA-NeXT and Qwen2.5-VL. **Most relevant paper for video reasoning RL.**
- **VersusQ** (arXiv:2605.21130, May 2026) — Margin-Coupled GRPO for video quality assessment. Jointly optimizes relational reasoning and continuous margin regression.
- **PreResQ-R1** (arXiv:2511.05393, Nov 2025) — Dual-branch reward formulation (intra-sample response coherence + inter-sample preference alignment) optimized via GRPO. SOTA on 10 IQA and 5 VQA benchmarks with only **6K images + 28K videos**.
- **VANS** (arXiv:2511.16669, Nov 2025) — Joint-GRPO to align VLM with video diffusion model for next-event prediction.

**Practical Note for RTX 4070:** GRPO requires multiple rollouts per training step (usually 8-16 responses). This means **8-16x the forward pass cost**. For RTX 4070:
- **Impossible** for 8B models at BF16 (VRAM blowup from storing N rollouts x sequence length)
- **Feasible with QLoRA** for 3B models if partial rollout batching is used (N=4, rollout size 512 tokens)
- **Workaround:** Offline GRPO — pre-compute rollouts, store on disk, then train on static preference pairs (essentially DPO with GRPO-generated data)

### 4.3 ORPO (Odds Ratio Preference Optimization)

**Status:** No dedicated video MLLM ORPO papers found. ORPO combines SFT and preference optimization in one stage. Since video MLLMs require separate vision-language alignment pre-training, pure ORPO is less applicable. However, it could work for the instruction tuning stage of a model that already has vision-language alignment.

**Recommendation:** Skip ORPO for video. Use two-stage SFT → DPO instead.

### 4.4 SimPO (Simple Preference Optimization)

**Status:** No dedicated video MLLM SimPO papers. SimPO simplifies DPO by using the average log-probability of the response (length-normalized) as implicit reward, removing the need for a reference model.

**Advantage for RTX 4070:** **No reference model** means ~2GB less VRAM compared to DPO. This is the most memory-efficient preference optimization method for consumer GPUs.

**Recommendation:** SimPO is potentially the best choice for RTX 4070 preference optimization. Reference-free nature saves critical VRAM. Needs experimental validation for video tasks.

### 4.5 Preference Optimization Summary

| Method | Reference Model? | VRAM vs SFT | Best For | RTX 4070 Score |
|--------|-----------------|-------------|----------|----------------|
| DPO | Yes (2x memory) | +50-100% | Well-established, reliable | ✅ Viable |
| SimPO | No (memory efficient) | +10-20% | Memory-constrained setups | ✅✅ Best choice |
| ORPO | No (integrated) | +5-10% | Combined SFT+PO | ⚠️ Untested for video |
| GRPO | No (needs rollouts) | +100-300% | Reasoning improvement | ⚠️ Offline only |

### 5.2 Dataset Strategy for RTX 4070

**Recommendation:** Use a tiered approach:

1. **Quick-start (1-2GB max):** CoTasks-enhanced NeXT-QA (~50K, ~200MB), V1-33K (~100MB)
2. **Domain SFT (5-10GB):** Streamo-Instruct-465K subsample (~50K for domain adaptation), ViCA-322K (~2GB)
3. **Preference data (500MB-2GB):** Generate using existing MLLM rollouts (self-play), MR²Bench-Video

### 5.3 Data Loading Considerations

For streaming video data during training on RTX 4070:
- **Pre-extract frames:** Extract 8-32 frames per video at 224/256px during dataset preparation, NOT during training
- **Store as .npz or .npy arrays:** Avoid loading full videos via decord during training
- **Use HuggingFace Datasets:** Memory-mapped access via `datasets.load_dataset()` with streaming mode
- **Recommended format:** Store pre-extracted frame embeddings (if using frozen vision encoder) or pre-extracted frame images

## 7. Feasibility for RTX 4070 12GB

### 7.1 Memory Budget Breakdown

| Component | 2B Model (BF16) | 3B Model (BF16) | 8B Model (QLoRA 4-bit) |
|-----------|-----------------|-----------------|------------------------|
| Model weights | ~4 GB | ~6 GB | ~5 GB |
| LoRA adapters (r=16) | ~50 MB | ~80 MB | ~200 MB |
| Optimizer states (AdamW) | ~8 GB | ~12 GB | N/A (4-bit does GaLore) |
| Activations (BS=1, seq=2048) | ~1 GB | ~1.5 GB | ~2 GB |
| **Total (LoRA SFT)** | **~13 GB** | **~20 GB** | **~7.5 GB** |
| **With gradient checkpointing** | **~8 GB ✅** | **~10 GB ⚠️** | **~6 GB ✅** |

**Key Finding:** BF16 LoRA SFT is **NOT feasible** for 3B+ models on 12GB VRAM with standard AdamW (optimizer states dominate). Must use:
- **paged_adamw_8bit** (from bitsandbytes) — reduces optimizer states to ~2GB for 3B
- **GaLore** (8-bit optimizer, compatible with LoRA) — further reduction
- **Adafactor** (memory-efficient optimizer) — works well with Transformers/LLaMA-Factory

### 7.2 Memory-Efficient Training Config (RTX 4070 12GB)

```yaml
# Memory saving measures (all can be combined)
optim: paged_adamw_8bit
gradient_checkpointing: true
gradient_checkpointing_kwargs:
  use_reentrant: false
fp16: true  # or bf16: true if GPU supports it
per_device_train_batch_size: 1
gradient_accumulation_steps: 8  # maintain effective batch size

# For LoRA
lora_rank: 8  # lower rank = less memory
lora_target_modules: ["q_proj", "v_proj"]  # fewer targets = less memory
```

### 7.3 Viable Training Configurations

| Config | Model | VRAM | Speed | Quality |
|--------|-------|------|-------|---------|
| **A** LoRA BF16, BS=1 | SmolVLM2-500M | ~4 GB | Fast | Good for simple tasks |
| **B** LoRA BF16, BS=4 | SmolVLM2-500M | ~6 GB | Fastest | Good for simple tasks |
| **C** LoRA BF16, BS=1 | SmolVLM2-2.2B | ~8 GB | Medium | Best for 2B class |
| **D** QLoRA 4-bit, BS=1 | VideoChat-Flash-2B | ~7 GB | Medium | Best for 2B video-native |
| **E** LoRA BF16, BS=1 (8-bit optim) | Qwen2.5-VL-3B | ~10 GB | Medium | Best captioning/QA |
| **F** QLoRA 4-bit, BS=1 | InternVideo3-8B | ~6 GB | Slow | Best overall quality |

**Recommendation:** Start with **Config C** (SmolVLM2-2.2B LoRA BF16) for rapid iteration, then **Config D** (VideoChat-Flash-2B LoRA BF16) for video-native architecture, and finally **Config F** (InternVideo3-8B QLoRA 4-bit) for production.

## Appendix: Quick-Start Script

```python
#!/usr/bin/env python3
"""Minimal LoRA SFT for SmolVLM2 on RTX 4070 (12GB) — video captioning."""

import torch
from peft import LoraConfig, get_peft_model, PeftModel
from transformers import (
    AutoModelForImageTextToText,
    AutoProcessor,
    TrainingArguments,
    Trainer,
)
from datasets import Dataset
import numpy as np
from PIL import Image

# 1. Load base model + processor
model = AutoModelForImageTextToText.from_pretrained(
    "HuggingFaceTB/SmolVLM2-500M-Video-Instruct",
    torch_dtype=torch.bfloat16,
    device_map="cuda",
)
processor = AutoProcessor.from_pretrained(
    "HuggingFaceTB/SmolVLM2-500M-Video-Instruct"
)

# 2. Apply LoRA
peft_config = LoraConfig(
    r=16,
    lora_alpha=32,
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
)
model = get_peft_model(model, peft_config)
model.print_trainable_parameters()  # Should show ~0.3-0.5% trainable

# 3. Prepare data (pre-extracted frames)
def collate_fn(batch):
    """Collate video frames + captions into model inputs."""
    texts = []
    images = []
    for item in batch:
        messages = [
            {
                "role": "user",
                "content": [
                    {"type": "image"},
                    {"type": "text", "text": item["instruction"]},
                ],
            },
            {"role": "assistant", "content": [{"type": "text", "text": item["caption"]}]},
        ]
        texts.append(processor.apply_chat_template(messages, add_generation_prompt=False))
        images.append([Image.fromarray(f) for f in item["frames"]])
    
    # Process in batch
    return processor(
        text=texts,
        images=images,
        padding=True,
        return_tensors="pt",
    )

dataset = Dataset.from_list([
    {
        "frames": [np.zeros((224, 224, 3), dtype=np.uint8)],  # Replace with real frames
        "instruction": "Describe this video scene",
        "caption": "A person walks across a room.",
    }
])

# 4. Configure training
training_args = TrainingArguments(
    output_dir="./smolvlm2-lora-output",
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,
    learning_rate=2e-4,
    warmup_steps=50,
    num_train_epochs=3,
    logging_steps=10,
    save_steps=500,
    fp16=True,
    gradient_checkpointing=True,
    dataloader_pin_memory=False,
    remove_unused_columns=False,
    report_to="none",
)

trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=dataset,
    data_collator=collate_fn,
)

# 5. Train
trainer.train()

# 6. Save adapter
model.save_pretrained("./smolvlm2-lora-adapter")
processor.save_pretrained("./smolvlm2-lora-adapter")

# 7. Load for inference
base = AutoModelForImageTextToText.from_pretrained(
    "HuggingFaceTB/SmolVLM2-500M-Video-Instruct",
    torch_dtype=torch.bfloat16,
    device_map="cuda",
)
model = PeftModel.from_pretrained(base, "./smolvlm2-lora-adapter")
```
