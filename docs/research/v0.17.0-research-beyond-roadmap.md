## 1. Autonomous Agentic Pipeline (Beyond Monolithic Processing)

### Current State
The pipeline runs as a linear sequence: extract audio → detect scenes → extract frames → YOLO → OCR → CLIP → X-CLIP → index. All stages are fixed, run on every video, and don't adapt to content.

### v0.17.0 Vision: Dynamic Stage Orchestration

**Instead of running every stage on every video, let an LLM/agent decide which stages to run based on video content analysis.**

```
Phase 1: Quick scan (30s analysis of 5 random frames)
  → Classify video type (lecture, vlog, sports, interview, movie, screen recording, etc.)
  → Decide which stages are relevant:
    - Screen recording: OCR high priority, transcription low
    - Music video: Transcription medium, objects low  
    - Interview: Diarization high, OCR low
    - Sports: Action recognition high, OCR low
    - Lecture: OCR + Transcription high, action recognition skip

Phase 2: Execute only selected stages (save GPU time, disk space, indexing cost)
Phase 3: Post-process with video-type-aware chunking and metadata
```

**Implementation approach:**
```python
class PipelineOrchestrator:
    """LLM-driven pipeline stage selection based on video content analysis."""
    
    async def analyze_video_type(self, video_path: str) -> VideoProfile:
        """Sample 5 frames, use SmolVLM2 500M to classify video type."""
        
    def build_plan(self, profile: VideoProfile) -> PipelinePlan:
        """Return only the relevant stages based on video type."""
```

**Estimated effort:** 3-5 days (new module + wiring into existing pipeline)
**Impact:** 30-50% faster processing, 40-60% less disk usage for stage outputs

## 2. Video Type Classification & Adaptive Processing

Building on the orchestrator concept, classify video types to optimize every parameter:

| Video Type | Frame Rate | Scene Detection | Key Models | Chunk Strategy |
|---|---|---|---|---|
| **Lecture/Webinar** | 0.1 fps | Low threshold | OCR → Transcription → MLLM | Large chunks (60s) |
| **Sports** | 1-2 fps | Medium threshold | Action → YOLO → MLLM | Small chunks (10s) |
| **Screen recording** | 0.05 fps | FFmpeg MVs | OCR → YOLO (UI elements) | Dynamic by app window |
| **Interview/Podcast** | 0.2 fps | Histogram | Diarization → Transcription → Face ReID | Scene-based |
| **Movie/Show** | 0.5 fps | Adaptive | All stages | Multi-granularity |
| **Vlog/Talking head** | 0.1 fps | Hash | Face → Transcription → OCR | Scene-based |

**Implementation:** Refine from the existing `_adaptive_frame_samples()` method in pipeline.py and add `video_type` config field.

**Estimated effort:** 2-3 days
**Impact:** Optimal quality-per-second for each video type

## 3. Face Recognition System (InsightFace Integration)

### Current State
YOLO detects "person" class with no identity. No face recognition exists.

### v0.17.0 Plan
Add **InsightFace** (MIT, ~1.5 GB VRAM) as an optional face recognition layer:

- **Detection:** RetinaFace (included with InsightFace) — detects faces at any angle
- **Recognition:** ArcFace (included) — 512-dim embedding per face
- **Clustering:** DBSCAN on face embeddings to group same person across scenes
- **Persistence:** Save face gallery to disk for cross-video person matching

```python

    def match_or_create(self, embedding: np.ndarray) -> str:
        """Match against gallery or create new person ID."""
        # Cosine similarity > 0.5 threshold for same person

        """DBSCAN cluster all embeddings -> {frame_timestamp: person_id}"""
```

**Integration point:** Pipeline Step 7.5 (after YOLO, before X-CLIP). Face metadata → ChromaDB → enables "find all scenes with [person_name]" queries.

**VRAM:** ~1.5 GB peak. Load/unload on demand per scene.
**Estimated effort:** 3-4 days

## 4. ChromaDB Upgrade to v3.x or Alternative Vector Store

### Current State
ChromaDB 1.x (latest stable). Works well but has limitations: no built-in metadata filtering beyond `where`, limited batch operations, no serverless embedded mode in v1.x.

### Research Findings
- **ChromaDB 3.x** (if released by late 2026) — major rewrite with native IA (ScaNN) support, improved metadata filtering, and multi-tenancy. Watch for breaking changes.
- **LanceDB** (0.15+) — columnar vector-store with zero-copy reads, GPU-accelerated ANN, S3-native storage. Growing fast in ML pipelines.
- **Milvus Lite** — embedded vector DB with same API as full Milvus. Fixes ChromaDB's scaling issues while remaining embeddable.
- **BGE-VL + FAISS** — pure FAISS ANN as primary with SQLite for metadata. Maximum control, fewest dependencies.

### Recommendation
**Stay with ChromaDB for now.** It works well, 1.5+ supports metadata filtering effectively, and the codebase is tightly coupled to its API. If performance becomes a bottleneck with >100 videos or >10K scenes, migrate to LanceDB (closest API match, columnar I/O).

**Estimated effort:** If migration needed — 2-3 days (mostly API translation in rag.py)

## 5. Pipeline Caching & Incremental Re-Indexing

### Current State
Pipeline always re-processes from scratch. No cache of intermediate results. If YOLO fails on frame 50, all 50 frames are re-detected.

### v0.17.0 Plan

**Checkpoint-based caching:**
```python
# Each stage checks for existing cache
def extract_audio(self, video_path) -> Path:
    cache_key = hash_file(video_path) + ":audio"
    if self.cache.exists(cache_key):
        return self.cache.load(cache_key)
    result = self._extract_audio_impl(video_path)
    self.cache.save(cache_key, result)
    return result
```

**Incremental re-indexing:**
```python
def index_video_incremental(self, video_id: str, new_scenes: List[SceneInfo]):
    """Only index scenes that changed, update existing ChromaDB entries by ID."""
    existing_ids = set(self.collection.get(where={"video_id": video_id})["ids"])
    new_ids = {f"{video_id}_scene_{s.scene_id:04d}" for s in new_scenes}
    to_delete = existing_ids - new_ids
    to_add = new_ids - existing_ids
    if to_delete:
        self.collection.delete(ids=list(to_delete))
    # Only add new/changed scenes
```

**Config fields:**
```python
pipeline_cache_enabled: bool = True
pipeline_cache_dir: Path = "data/cache"
```

**Impact:** 70-90% faster re-processing when only config changes or partial re-runs.

## 6. Gradio 6 Workflow & MCP Server Deep Integration

### Current State
Gradio 6.19.0 introduced Workflow subgraph API. The project has a 4-phase integration plan.

### v0.17.0 Extension: Full MCP Server

Beyond the planned Gradio Workflow integration, expose each pipeline stage as **MCP tools** so that Hermes Agent (or any MCP client) can call individual pipeline stages:

```python
# mcp_tools.py
@mcp.tool()
async def extract_frames(video_path: str, frame_rate: float = 0.5) -> dict:
    """Extract frames from a video at the given frame rate."""

@mcp.tool()
async def detect_scenes(video_path: str, detector: str = "adaptive") -> list:
    """Detect scene boundaries in a video."""

@mcp.tool()
async def transcribe_audio(video_path: str) -> dict:
    """Transcribe audio from a video using faster-whisper."""

@mcp.tool() 
async def detect_objects(frame_paths: list, model: str = "yolo26x.pt") -> list:
    """Run YOLO object detection on a list of frame images."""

@mcp.tool()
async def search_video(query: str, video_id: str = None, top_k: int = 5) -> list:
    """Search indexed video content with natural language query."""

@mcp.tool()
async def get_video_library() -> list:
    """List all indexed videos in the library."""
```

This allows agentic workflows like:
```
User: "Find all scenes with a person in a red shirt and transcribe what they say"
→ MCP tool: search_video("person in red shirt") → get timestamps
→ MCP tool: extract_frames(timestamps) → get frames with person 
→ MCP tool: transcribe_audio(timestamp_range) → get speech
→ LLM: synthesize answer
```

**Implementation:** Create `video_analysis/mcp_tools.py` using Hermes MCP SDK. Each tool wraps an existing pipeline method.

**Estimated effort:** 2-3 days
**Impact:** Enables external agents to use video analysis as a composable tool library

## Design Decisions

### Why NOT to use certain approaches

| Approach | Rejected Because |
|----------|-----------------|
| InternVideo2.5 (8B) | 16 GB VRAM > 12 GB budget |
| RAFT optical flow | 1.5 GB VRAM + 100ms/frame vs FFmpeg MVs (0 GB + <1ms) |
| Neo4j graph DB | Overkill for <1000 scene nodes; NetworkX is sufficient |
| Celery/RabbitMQ workers | Overkill for single-GPU pipeline; asyncio + FastAPI background tasks suffice |
| Kubernetes | Overkill for single-node self-hosted deployment; Docker Compose is sufficient |
| Gemini API / cloud VLMs | Project is 100% local by design; no API keys allowed |
| LanceDB migration now | ChromaDB works fine; migration effort > benefit at current scale |

### Why TO adopt these approaches

| Approach | Why |
|----------|-----|
| ByteTrack via Ultralytics (not BoxMOT) | AGPL-3.0 license concern; Ultralytics trackers are built-in, zero extra deps |
| FFmpeg motion vectors | Zero GPU cost, always available for h264/h265, <1ms per frame |
| InsightFace | MIT license, best open-source face recognition, well-maintained |
| SmolVLM2 500M for video classification | Only 0.5-1 GB VRAM, can run alongside pipeline |
| MCP tool server | Enables agentic workflows; Hermes + any MCP client can call individual stages |
