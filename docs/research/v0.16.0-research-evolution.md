## 1. Entity-Level Tracking Across Scenes (Persistent IDs)

### The Gap

Currently, YOLO detects objects per-frame with **no persistent identity**. A person in
scene 1 has no connection to the same person in scene 3 — they're just two independent
detection boxes. Scene-graph entity edges use text-parsed object names (e.g., "person")
which conflates all instances of the same class.

### Technology Candidates

| Technology | Type | License | VRAM | Description |
|-----------|------|---------|------|-------------|
| **ByteTrack** | Multi-Object Tracking | MIT | Minimal | SOTA online MOT, YOLO-native, no re-ID model needed |
| **BoT-SORT** | Multi-Object Tracking | MIT | Minimal | Better occlusion handling, camera motion compensation |
| **BoxMOT** | Aggregator package | MIT | Minimal | Unified API for ByteTrack + BoT-SORT + DeepOCSORT + ImprAssocFlow |
| **DeepSORT** | Multi-Object Tracking | GPL-3.0 | ~0.5 GB | Cosine appearance matching + Kalman filter |
| **ViT-ReID** | Re-identification | MIT | ~1 GB | Vision Transformer person ReID, cross-camera matching |
| **InsightFace** | Face Recognition | MIT | ~1.5 GB | ArcFace face detection + recognition, tracks people by face ID |
| **YOLO OBB** | Oriented Object Tracking | AGPL-3.0 | ~1 GB | YOLO-oriented bounding boxes via BoxMOT |

**Recommendation**: **ByteTrack via BoxMOT** — integrates directly with YOLOv26
already in the pipeline, no re-ID model needed for basic persistent IDs, minimal VRAM
(~500 MB shared with YOLO). For person-specific tracking across long gaps, **InsightFace**
(Swag-API) adds face recognition and clustering.

### Integration Strategy

```
pipeline.py:
  Step 7 (YOLO detection):
    - After per-frame YOLO, run ByteTrack on sequential frames
    - Assign persistent track_id to each detection
    - Store track_ids in FrameInfo.objects[{..., "track_id": int}]

models.py:
  - Add tracking_id: Optional[int] to FrameInfo
  - Add tracked_entities: List[TrackedEntity] to SceneInfo

rag.py (indexing):
  - Index entity presence per scene:
    - TrackedEntity(entity_id, label, confidence, first_seen, last_seen, frames)
  - Add entity-level metadata to Chroma chunks

scene_graph.py:
  - Upgrade entity-shared edges to use track_ids instead of text-parsed names
  - Cross-scene edges by same entity_id → real person persistence
```

### Key Insight

ByteTrack produces track_ids consistent within a single video run. For cross-video
entity matching, a separate ReID step or face clustering is needed — that aligns with
the next roadmap item (cross-video scene graph edges).

## 2. Cross-Video Scene Graph Edges

### Current State

`scene_graph.py` builds edges **per video** — temporal edges only connect scenes within
the same video. The graph adjacency dict `{(video_id, scene_id): set(...)}` already
supports cross-video keys but doesn't build edges between different videos.

### Target

Enable semantic connections between scenes across different videos, so a query like
"find all interviews in any video" retrieves relevant scenes from the entire library.

### Technology

| Approach | Description | Pros | Cons |
|---------|-------------|------|------|
| **BGE-VL cross-video similarity** | Embed all scene descriptions via BGE-VL, cluster using FAISS/HDBSCAN | Leverages existing BGE-VL, no new deps | O(n²) for n scenes |
| **Entity-based matching** | Connect scenes across videos sharing the same entity_id (from tracking) or face ReID | Natural extension of entity tracking | Needs entity tracking first |
| **Semantic keyword clusters** | Scene text Jaccard/tf-idf across video boundaries | Zero deps, fast | Only text, misses visual similarity |
| **LLM-generated cross-links** | Use SmolVLM2 to identify cross-video thematic connections | Deep semantic understanding | Expensive, slow for large libraries |

**Recommendation**: **Phase 1** — Extend scene_graph.py's `rebuild()` to compute
semantic Jaccard + entity-based edges across ALL videos (not just within same video_id).
This leverages the existing code path — the adjacency structure already separates by
`(video_id, scene_id)` tuples. **Phase 2** — Add BGE-VL cross-video scene embedding
comparison via FAISS approximate kNN (scalable to 10K+ scenes).

### Integration Strategy

```python

        k1, k2 = keys_list[i], keys_list[j]
```

### BGE-VL Cross-Video Scene Similarity

Since BGE-VL is already loaded during RAG indexing, scene embedding vectors can be
computed during `index_video()` and stored as ChromaDB metadata for batched comparison.
Estimated: 100ms per scene embedding, so 100 scenes = 10 seconds total. FAISS IVF index
reduces cross-comparison from O(n²) to O(n log n).

## 3. Gradio 6 Workflow Integration

### Gradio 6.19+ Workflow API

Gradio 6.19.0 (June 2026) introduced **Workflow subgraphs** — composable API endpoints
where each subgraph is exposed as a named endpoint via `/info`, `/call`, and `/api`,
with a "View API" panel in the workflow canvas. Key features:

- **Subgraph endpoints**: Each workflow node becomes an independently callable API
- **`gr.Workflow` blocks**: Define composable pipeline stages as reusable subgraphs
- **MCP integration**: Gradio 6 can expose workflow stages as MCP tools
- **`/api/{workflow_name}/{subgraph_name}`**: Direct HTTP access to sub-stages

### Architecture

Currently, the UI triggers the entire pipeline monolithically:

```python
# ui/app.py — current
process_btn.click(
    fn=run_pipeline_wrapper,
    inputs=[video_input, url_input],
    outputs=[progress, chat, video_display, ...]
)
```

**Target architecture** (Gradio Workflow subgraphs):

```python
# Each pipeline stage as a Gradio workflow subgraph:
extract_audio_wf = gr.Workflow(
    fn=extract_audio_stage,
    inputs=gr.Video(),
    outputs=[gr.Audio(label="Extracted Audio"), gr.State(meta)]
)

detect_scenes_wf = gr.Workflow(
    fn=detect_scenes_stage,
    inputs=[gr.Video(), gr.State(meta)],
    outputs=[gr.JSON(label="Scenes"), gr.State(meta)]
)

# Compose into full pipeline
full_pipeline = gr.Workflow(
    fn=run_full_pipeline,
    inputs=[gr.Video(), gr.Textbox(label="URL")],
    outputs=[gr.JSON(label="Results")],
    subworkflows=[extract_audio_wf, detect_scenes_wf, ...]
)
```

### FastAPI Hybrid Approach

A practical alternative (or complement): expose each pipeline stage via **FastAPI**
endpoints (the project already has a `/health` endpoint). The FastAPI routes can be
auto-generated from pipeline step methods:

```python
# FastAPI endpoints per stage
POST /api/video/{id}/extract-audio
POST /api/video/{id}/detect-scenes
POST /api/video/{id}/transcribe
POST /api/video/{id}/detect-objects
POST /api/video/{id}/describe-scenes
POST /api/video/{id}/index
GET  /api/video/{id}/status
```

### MCP Server Integration

Gradio 6's MCP integration means each pipeline workflow stage could also be exposed
as an MCP tool — allowing external agents (like Hermes subagents) to call individual
pipeline stages. This enables distributed processing where different workers handle
different pipeline stages.

### Recommendation

- **Phase A**: Refactor pipeline.py's monolithic `process_video()` into individual
  stage methods that can be called independently (already partially done — each step
  is a method)
- **Phase B**: Expose stages as FastAPI endpoints (existing `FastAPI app` in ui/app.py
  can be extended with routers)
- **Phase C**: Add Gradio Workflow subgraphs for composable UI pipelines
- **Phase D**: Add MCP tool definitions for each stage

## 4. Sparse-Frame Optical Flow

### Current Adaptive Sampling

The existing `_adaptive_frame_samples()` uses a simple cosine-based density function:
3x density in the first/last 10% of each scene, base rate elsewhere. This is
**motion-unaware** — it doesn't actually measure motion, it just assumes scene
boundaries are where motion happens.

### Technology Candidates

| Technology | Type | License | VRAM | Speed | Description |
|-----------|------|--------|------|------|-------------|
| **OpenCV Farneback** | Dense optical flow | Apache 2.0 | CPU | ~30ms/frame (480p) | Classic GPU-less option, good for low-res |
| **OpenCV DIS** | Dense optical flow | Apache 2.0 | CPU | ~15ms/frame (480p) | Fast alternative to Farneback |
| **RAFT** | Deep flow (ECCV 2020) | BSD-3 | ~1.5 GB | ~100ms/frame | SOTA accuracy, heavy |
| **GMFlow** | Deep flow (CVPR 2022) | MIT | ~1 GB | ~80ms/frame | Better speed/accuracy than RAFT |
| **FlowFormer** | Deep flow (CVPR 2023) | MIT | ~2 GB | ~200ms/frame | Highest accuracy transformer-based |
| **FFmpeg motion vectors** | Macroblock vectors | LGPL | CPU | <1ms/frame | Extracts existing h264/h265 motion vectors — FREE! |

### Key Insight: FFmpeg Motion Vectors

The most practical approach for a pipeline that already uses FFmpeg is to **extract
existing motion vectors from the video codec**. Every h264/h265 video already encodes
per-macroblock motion vectors as part of compression. FFmpeg can dump these with:

```bash
ffprobe -v quiet -select_streams v -show_entries frame=pkt_pts_time,\
  side_data_list,mb_type -of csv video.mp4
```

Or extract motion vector visualization:

```bash
ffmpeg -flags2 +export_mvs -i video.mp4 -vf codecview=mv=pf+bf+bb \
  motion_vectors.mp4
```

This costs **zero GPU** and takes **<1ms per frame** — the data is already computed
by the video encoder. The motion vector magnitude across all macroblocks gives a direct
per-frame "motion score" that can guide adaptive sampling.

## 5. Cutting-Edge Developments (Feb–Jun 2026)

### Video Understanding Models

| Model | Released | Params | VRAM | Key Feature |
|-------|----------|--------|------|-------------|
| **SmolVLM2** (HuggingFace) | Mar 2026 | 256M/500M/2.2B | 0.5-5.2 GB | Apache 2.0, transformers-native, 3 model sizes |
| **VideoChat-Flash** (OpenGVLab) | Dec 2025 | 2B | ~5.4 GB | MIT, ICLR 2026, 16 tok/frame, MVBench 70.0 |
| **InternVideo2.5** (OpenGVLab) | Feb 2026 | 8B | ~16 GB | Too heavy for 12GB, replaced VideoMAE |

### Key Trends

1. **Small video MLLMs**: The trend is toward models under 3B that fit consumer GPUs.
   SmolVLM2's 500M variant (Apache 2.0, ~1-2 GB VRAM) can run alongside the pipeline.

2. **Graph-based video RAG**: VGent (NeurIPS 2025 Spotlight) and ViG-RAG (AAAI 2026)
   established graph structures for video retrieval. The scene_graph.py module
   already implements this pattern.

3. **Agentic RAG**: Multi-round retrieval with confidence checking (implemented in
   v0.15.0) — the project is already ahead of most open-source tools on this front.

4. **Gradio as API platform**: Gradio 6.19+ Workflow subgraphs and MCP integration
   make it a full API platform, not just a UI library.

5. **FFmpeg codec analysis**: Using compressed-domain features (motion vectors,
   macroblock types) for zero-cost video analysis is an emerging pattern.

### What Hasn't Changed

- **ChromaDB** remains the best self-hosted vector DB for this use case
- **BGE-VL** still best multimodal embedder under 1 GB VRAM
- **faster-whisper** still fastest local transcription with int8 quantization
- **YOLO** has iterated to v26 but the pattern hasn't changed
- **PySceneDetect** still the gold standard for scene detection
