# Cross-Video Scene Graph Edges for Multi-Video Semantic Retrieval

> **Date:** 2026-06-26
> **Context:** Extending the existing per-video scene graph (`scene_graph.py` v0.14.0) to support cross-video edges for multi-video semantic retrieval.
> **Current stack:** ChromaDB + BGE-VL + NetworkX scene graph (per-video temporal/entity/semantic edges)
> **Hardware constraint:** 12GB VRAM

## 1. VGent — NeurIPS 2025 Spotlight (arXiv:2510.14032)

### Paper Details

- **Title:** "VGent: Graph-based Retrieval-Reasoning-Augmented Generation For Long Video Understanding"
- **Authors:** Xiaoqian Shen, Wenxuan Zhang, Jun Chen, Mohamed Elhoseiny (KAUST + Meta AI)
- **Venue:** NeurIPS 2025 (Spotlight)
- **arXiv:** [2510.14032](https://arxiv.org/abs/2510.14032)
- **Project page:** https://xiaoqian-shen.github.io/Vgent/
- **GitHub:** https://github.com/xiaoqian-shen/Vgent (48 stars, 7 forks)
- **Results:** +3.0-5.4% over base LVLMs on MLVU, +8.6% over SOTA video RAG methods

### Architecture (4 Stages)

```
1. Offline Video Graph Construction
   ├── Partition video into clips (K frames each, default 64)
   ├── Visual Entity Extraction via LVLM (objects, subjects, actions per clip)
   ├── Graph Construction (nx.Graph)
   │   └── Nodes = video clips (indexed by position)
   │   └── Edges = entity co-occurrence (same entity name = edge)
   └── Entity Merging: deduplicates via BGE text embedding similarity >0.7

2. Graph-Based Retrieval
   ├── Keyword extraction from user query via LVLM
   ├── BGE-large-en-v1.5 text similarity between keywords and node content
   └── Top-K nodes retrieved (default K=20)

3. Structured Reasoning
   ├── Decompose question into verification sub-queries
   ├── For each retrieved node → run sub-queries → confidence check
   ├── Filter to top-N refined nodes (default N=5)
   └── Aggregate information across verified clips

4. Multimodal Augmented Generation
   ├── Concatenate refined clips (frames + subtitles)
   └── Feed to LVLM for final answer generation
```

### Graph Construction Algorithm (Core Innovation)

```python
# Simplified from VGent source
def construct_graph(video_inputs, subtitles):
    video_graph = nx.Graph()
    entity_graph = defaultdict(set)  # entity_name -> set of clip indices
    
    for clip_idx in range(num_clips):
        # 1. Extract entities via LVLM (person, object, action, scene)
        entities = generate_entities(entity_prompt, clip_video)
        
        for entity_name, entity_desc in entities:
            # 2. Deduplicate: compare with existing entities via BGE embedding
            best_sim = 0.0
            most_similar = None
            for existing_name in entity_graph:
                sim = cosine_similarity(
                    embed(entity_name), 
                    embed(existing_name)
                )
                if sim > best_sim:
                    best_sim = sim
                    most_similar = existing_name
            
            # 3. Either merge with existing entity or create new one
            if best_sim > 0.7:
                entity_graph[most_similar].add(clip_idx)
                video_graph.add_edge(clip_idx, existing_idx, label=most_similar)
            else:
                entity_graph[entity_name].add(clip_idx)
    
    return video_graph, entity_graph
```

### Key Insights for Cross-Video Application

1. **VGent is per-video, not cross-video**: The graph is built for a single long video. Each node is a clip within that video. No mechanism exists for linking across different videos.

2. **Entity deduplication via embedding similarity is directly transferable**: The same BGE embedding approach can merge entities across videos.

3. **The LVLM-based entity extraction is heavy**: Requires running an LVLM on every clip. For cross-video, the existing pipeline's YOLO + X-CLIP + OCR + transcript outputs can serve as a lighter alternative.

4. **NetworkX is already used**: VGent uses `networkx.Graph` — same library already in use by the project.

5. **8 A100 GPUs recommended**: VGent uses torchrun with multi-GPU. This is impractical for 12GB VRAM. The lightweight entity extraction approach is essential.

## 2. ViG-RAG — AAAI 2026

### Details from Research

- **Status:** Referenced across multiple research docs as AAAI 2026, ranked #6
- **Full title:** "ViG-RAG: Video-aware Graph Retrieval-Augmented Generation via Temporal and Semantic Hybrid Reasoning"
- **Verified publication:** AAAI 2026

### Key Contributions

| Component | Description |
|-----------|-------------|
| **Temporal-before edges** | Scene A → Scene B (sequential ordering) |
| **Temporal-overlap edges** | Scene A ⟷ Scene C (overlapping frame ranges) |
| **Entity co-occurrence edges** | Both scenes mention/detect same entity |
| **Visual similarity edges** | High CLIP/BGE-VL embedding cosine similarity |
| **Action transition edges** | Connected by action continuity (X-CLIP) |

### Relevance to Cross-Video

ViG-RAG's **entity co-occurrence** and **visual similarity** edges are the natural candidates for cross-video linking — they don't depend on temporal proximity. The existing `scene_graph.py` already implements these three edge types (`temporal`, `entity_shared`, `semantic_similar`) for per-video graphs. The extension to cross-video requires:

1. Removing the per-video isolation of entity names
2. Adding cross-video semantic similarity computation
3. Introducing a global entity registry

## 3. VideoRAG — KDD 2026 (arXiv:2502.01549)

### Paper Details

- **Title:** "VideoRAG: Retrieval-Augmented Generation with Extreme Long-Context Videos"
- **Authors:** Xubin Ren, Lingrui Xu, Long Xia, Shuaiqiang Wang, Dawei Yin, Chao Huang (HKU + Baidu)
- **Venue:** KDD 2026
- **arXiv:** [2502.01549](https://arxiv.org/abs/2502.01549)
- **GitHub:** https://github.com/HKUDS/VideoRAG (⭐ 3.1k stars, 433 forks)

### Core Architecture

```
Dual-Channel Architecture:

Channel 1: Graph-based Textual Knowledge Grounding
  └── Cross-video semantic relationships via Knowledge Graph
  └── Entities extracted from transcripts + captions
  └── Graph built across ALL videos, not per-video
  └── NetworkX-based graph storage (NetworkXStorage)
  └── Leiden clustering for community detection
  └── Node2Vec for node embeddings

Channel 2: Multi-Modal Context Encoding
  └── Visual features from video segments
  └── NanoVectorDB for video segment feature storage
  └── Segment-level captioning via MiniCPM-V-2.6

Query Mode: "videorag"
  └── Entity retrieval + graph walk + visual feature matching
  └── Multi-modal fusion for final answer
```

### Cross-Video Graph Implementation (Directly Relevant)

VideoRAG is the **only** paper that explicitly handles cross-video graphs. Three key design choices:

#### a) Global Entity Extraction

```python
# From VideoRAG's _op.py
# Entities extracted via LLM from ALL video segments across all videos
# Entity types: person, organization, location, event, object, concept
# Graph nodes = entities (not clips/scenes)
# Graph edges = relationships between entities
```

**Crucial difference from VGent:** VideoRAG builds an **entity-centric graph** (entities as nodes, co-occurrence as edges), while VGent builds a **clip-centric graph** (clips as nodes, shared entities as edges).

#### b) NetworkXStorage with GraphML Persistence

```python
# From gdb_networkx.py — directly reusable
class NetworkXStorage(BaseGraphStorage):
    """NetworkX-based graph storage with GraphML persistence."""
    
    @staticmethod
    def load_nx_graph(file_name) -> nx.Graph:
        if os.path.exists(file_name):
            return nx.read_graphml(file_name)
        return None

    @staticmethod
    def write_nx_graph(graph: nx.Graph, file_name):
        nx.write_graphml(graph, file_name)
    
    # Full API: upsert_node, upsert_edge, has_node, has_edge,
    #           get_node, get_edge, get_node_edges, node_degree,
    #           clustering (Leiden), embed_nodes (Node2Vec)
```

#### c) Entity-Centric vs Scene-Centric Design Decision

| Aspect | VGent (per-video) | VideoRAG (cross-video) |
|--------|------------------|----------------------|
| Node type | Video clip | Entity (person/object/concept) |
| Edge type | Entity co-occurrence | Semantic relationship |
| Graph scope | 1 video | All videos |
| Entity linking | No (single video) | Yes (global dedup) |
| LLM usage | Heavy (per-clip) | Moderate (per-chunk) |
| Storage | Pickle | GraphML |

### Hardware Requirements

VideoRAG claims: **Single NVIDIA RTX 3090 (24GB)** for processing hundreds of hours of video. With 12GB VRAM, the visual channel would need to be scaled down, but the graph channel (entity extraction + NetworkX) is lightweight enough.

## 4. Approach for Extending Per-Video Scene Graph to Cross-Video

### Current State (v0.14.0-v0.15.0)

The existing `scene_graph.py` already has:

```python
# Three edge types (per-video only):
# 1. Temporal: scenes adjacent in time (same video)
# 2. Entity_shared: scenes sharing YOLO objects, actions, people
# 3. Semantic_similar: scenes with similar text content

# K-hop expansion: traverses graph edges to find related scenes
# Lazy rebuild: from ChromaDB metadata on first query
```

### What Blocks Cross-Video Edges

Looking at the current code in `scene_graph.py`:

1. **No global entity registry**: Entity names are per-scene, never deduplicated across videos
2. **Semantic edges use word Jaccard similarity** (not embedding-based), which is video-specific
3. **K-hop expansion is scene-aware but not video-aware** — the adjacency dict stores `(video_id, scene_id)` tuples and does traverse across videos already, but cross-video edges are never *built*
4. **`k_hop_expand()` already handles cross-video**: The function works on `(video_id, scene_id)` tuples and has no video boundary check. The issue is that no cross-video edges exist to traverse.

### Proposed Approach: Two-Phase Extension

#### Phase 1: Global Entity Registry (Lightest)

Add a cross-video entity deduplication step during `rebuild()`:

```python
def rebuild(self):
    # ... existing per-video graph building ...
    
    # NEW: Cross-video entity linking
    # 1. Collect all entity names across all videos
    # 2. Embed them with BGE (already available in pipeline)
    # 3. Cluster/deduplicate entities with similarity > 0.85
    # 4. Update entity references in scene metadata
    # 5. Add cross-video entity_shared edges
    
    # NEW: Cross-video semantic edges
    # 1. Compute BGE embeddings for all scene documents
    # 2. Find cross-video scene pairs with similarity > threshold
    # 3. Add semantic edges with weight = similarity score
```

This reuses the **BGE embedding model already in the pipeline** and requires no new models.

#### Phase 2: Entity-Centric Graph (Stronger)

Add entity nodes to the graph (inspired by VideoRAG):

```
┌─────────────────────────────────────────────┐
│          CROSS-VIDEO GRAPH SCHEMA            │
│                                              │
│  Node types:                                 │
│    SCENE   — (video_id, scene_id) [existing] │
│    ENTITY  — (entity_id, name, type) [NEW]   │
│                                              │
│  Edge types:                                 │
│    SCENE ──has_entity──→ ENTITY  [NEW]       │
│    SCENE ──next_scene──→ SCENE   [existing]  │
│    SCENE ──semantic──→ SCENE    [existing]   │
│    ENTITY ──co_occurs──→ ENTITY  [NEW]       │
│                                              │
│  Cross-video retrieval:                      │
│    Query → find entity → all scenes with     │
│    that entity across ALL videos              │
└─────────────────────────────────────────────┘
```

### Concrete Implementation Plan

```python
# New methods for scene_graph.py

def _build_cross_video_entity_edges(self):
    """Link scenes across videos via shared entities."""
    # Group scenes by entity (existing extracted entities)
    entity_to_scenes = defaultdict(set)
    for (vid, sid), meta in self._node_meta.items():
        entities = self._extract_entities(meta)
        for ent in entities:
            entity_to_scenes[ent].add((vid, sid))
    
    # Add edges between any scenes sharing an entity
    # (already partially done via entity_shared within same video)
    for entity, scenes in entity_to_scenes.items():
        scenes_list = list(scenes)
        for i in range(len(scenes_list)):
            for j in range(i + 1, len(scenes_list)):
                s1, s2 = scenes_list[i], scenes_list[j]
                if s1[0] != s2[0]:  # Different videos
                    self._add_edge(s1, s2)

def _build_cross_video_semantic_edges(self):
    """Link scenes across videos via embedding similarity."""
    # Get all scene node keys grouped by video
    video_scenes = defaultdict(list)
    for key in self._node_meta:
        video_scenes[key[0]].append(key)
    
    video_ids = list(video_scenes.keys())
    
    # Compare scenes from different videos
    for i in range(len(video_ids)):
        for j in range(i + 1, len(video_ids)):
            scenes_a = video_scenes[video_ids[i]]
            scenes_b = video_scenes[video_ids[j]]
            
            # Sample or compute pairwise similarity
            # Use BGE embedding of scene document text
            for sa in scenes_a:
                for sb in scenes_b:
                    sim = self._compute_scene_similarity(sa, sb)
                    if sim > self.entity_similarity_threshold:
                        self._add_edge(sa, sb)
```

## 6. Lightweight Approaches (No Graph DB — Pure Python)

### Option A: ChromaDB-Only Cross-Video (No NetworkX Expansion)

For minimal changes, cross-video retrieval can work **without graph edges**:
1. Store `entities` as a metadata field in ChromaDB
2. During retrieval, search for query keywords in entity metadata
3. ChromaDB's built-in `$contains` filter can find scenes with matching entities across videos
4. No graph traversal needed

**Pro:** Zero new code, works today
**Con:** No K-hop expansion, no indirect relationships

### Option B: NetworkX Entity Graph (Recommended)

Build a **separate entity graph** (not replacing the scene graph):
- Nodes = unique entities across all videos (deduplicated via BGE embedding)
- Edges = co-occurrence in same scene (weighted by frequency)
- Cross-video query: entity → find all scenes across videos
- Scene graph + entity graph = full hybrid

**RAM estimate:** ~50-100 MB for 10K entities across 100 videos
**Build time:** ~2-5 seconds for entity dedup of 100 videos

### Option C: Lightweight Entity Index (Dict + Set)

For very small scale (< 10 videos, < 500 scenes):

```python
# Simple inverted index from entity → scenes
entity_index: Dict[str, Set[Tuple[str, int]]] = defaultdict(set)

# Scene → entities (already in ChromaDB metadata)
scene_entities: Dict[Tuple[str, int], Set[str]] = {...}

# Cross-video query
def find_scenes_by_entity(entity: str) -> List[Tuple[str, int]]:
    return list(entity_index.get(entity, set()))
```

**RAM:** Negligible (< 10 MB)

### Comparison Table

| Approach | Implementation Effort | Cross-Video | K-Hop | RAM | Accuracy |
|----------|---------------------|-------------|-------|-----|----------|
| ChromaDB-only | 0 hours | ✅ Basic | ❌ | 0 MB | Low |
| Dict + Set index | 1 hour | ✅ Simple | ❌ | <10 MB | Medium |
| **NetworkX entity graph** | **4-6 hours** | **✅ Full** | **✅** | **50-100 MB** | **High** |
| Neo4j/FalkorDB | 8+ hours | ✅ Full | ✅ | 1-4 GB | High |

**Recommendation:** NetworkX is the sweet spot for 12GB VRAM constraint and current scale (< 100 videos).
