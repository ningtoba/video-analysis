## 1. Video Storage & Compression Optimization

### Problem Statement

Currently, the pipeline extracts full-resolution frames (typically 1920×1080 or 1280×720) for every scene, saving them as JPEG/PNG to disk. For a 10-minute video at 0.5 fps, that's ~300 frames at ~200-500 KB each = 60-150 MB per video just for frames. With a multi-video library, this can grow to 10s of GB.

### Research Findings

| Approach | Compression Ratio | Quality | CPU/GPU | Notes |
|----------|------------------|---------|---------|-------|
| JPEG 85% (current) | 10:1 | Good | CPU | Standard, adequate |
| JPEG XL | 15-20:1 | Excellent | CPU | ~2× JPEG with better quality. libjxl available |
| WebP Lossy | 15:1 | Good | CPU | Browser-native, -40% size vs JPEG |
| AVIF | 20:1 | Excellent | CPU/GPU | Best compression, slowest encode |
| MozJPEG | 12:1 | Good | CPU | Better quality at same size as JPEG |
| PNG (lossless) | 2:1 | Perfect | CPU | Only for frames needing OCR/YOLO precision |
| Half-resolution thumbnails | 4:1 (area) | Acceptable | CPU | 960×540 for CLIP/analysis; full-res only for OCR |

### Recommendation: Tiered Frame Storage

```
frame_storage_mode = "tiered"  # or "full" (default), "thumbnails_only", "compressed"

When tiered:
  - Analysis: 960×540 JPEG 85% → ~50-80 KB per frame
  - OCR/YOLO: original resolution JPEG 90% → ~200-400 KB per frame  
  - Timeline: 320×180 WebP 80% → ~15-25 KB per frame
  - Archive: JPEG XL (after pipeline completes, during idle)

Estimated savings: 60-150 MB → 15-40 MB per 10-min video (-60-75%)
```

### Config Fields

```python
frame_storage_mode: str = "tiered"       # full, tiered, thumbnails_only, compressed
frame_analysis_size: int = 960           # Longest edge for analysis frames
frame_thumbnail_size: int = 320          # Longest edge for timeline thumbnails
frame_compression: str = "jpeg"          # jpeg, webp, avif, jpegxl
frame_compression_quality: int = 85      # 1-100
```

### Implementation Impact

- Minimal code changes — mostly in `pipeline.py` frame extraction + sprite sheet generation
- New dependency: `Pillow>=11.0` (already installed, supports WebP natively)
- Optional: `libjxl` for JPEG XL (add to Dockerfile)
- Zero VRAM impact (CPU-only)

## 2. Multi-Modal Conversation Memory for Video RAG

### Problem Statement

Current conversation memory is per-session only — when the user asks a question about a video, then later asks a follow-up about a different video, the chat interface has no persistent memory of the cross-video conversation context. Each query to ChromaDB is independent.

### Research: Beyond Simple Chat History

| Approach | Persistence | Cross-Video | Query Enhancement | Implementation |
|----------|-------------|-------------|-------------------|---------------|
| Session chat history (current) | In-memory | No | None | Simple list in chat.py |
| Episodic memory buffer | Per-session file | No | Recent Q&A as context prefix | ~50 lines, JSONL file |
| ChromaDB conversation store | Persistent | Yes | Previous Q&A as retrieval context | ~150 lines, separate collection |
| LongLive-RAG (arXiv) | Persistent | Yes | Searchable conversation memory | ~300 lines, metadata-filtered |
| Vector-expanded queries | Persistent | Yes | Embeds previous Q&A for context | ~200 lines, cross-collection search |

### Recommendation: ChromaDB Conversation Store

```python
class ConversationMemory:
    """Persistent conversation memory stored in a separate ChromaDB collection."""
    
    def __init__(self, config):
        self.collection = self._get_or_create_collection("conversation_memory")
        self.max_memories = 50  # Keep last 50 Q&A pairs
    
    def add_interaction(self, query: str, answer: str, video_id: str, timestamp: float):
        """Store a Q&A pair with video context."""
        embedding = self._embed(f"Q: {query}\nA: {answer}")
        self.collection.add(
            ids=[f"conv_{timestamp}"],
            embeddings=[embedding],
            metadatas=[{
                "video_id": video_id,
                "timestamp": timestamp,
                "type": "conversation"
            }],
            documents=[f"User asked: {query}\nAssistant answered: {answer}"]
        )
    
    def get_relevant_context(self, query: str, top_k: int = 3) -> List[str]:
        """Retrieve relevant past Q&A pairs for current query context."""
        results = self.collection.query(
            query_texts=[query],
            n_results=top_k,
        )
        return results["documents"][0] if results["documents"] else []
```

### Key Design Decisions

- **Separate collection** (`conversation_memory`) from video index — avoids polluting video search with conversation artifacts
- **Metadata filtering** — `video_id` field enables per-video conversation context
- **Size limit** — max 50 entries prevents ChromaDB bloat; older entries evicted via TTL
- **Embedding model** — reuses existing `BGE-VL-base` to avoid extra VRAM

### Config Fields

```python
conversation_memory_enabled: bool = True
conversation_memory_max_entries: int = 50
conversation_memory_ttl_days: int = 30
```

### Integration Point

In `chat.py`:
1. Before querying ChromaDB, retrieve relevant conversation memories
2. Prepend conversation context to the LLM prompt
3. After answering, store the Q&A pair

**Estimated effort:** 1 day
**Impact:** Enables natural follow-up questions like "what about the other video?" or "show me more clips like that"

## 3. Real-Time / Streaming Video Analysis

### Problem Statement

The current pipeline is batch-oriented — it processes a complete video file. For live streams, CCTV feeds, or incremental monitoring, a real-time/streaming mode is needed.

### Research: Streaming Patterns

| Approach | Latency | Memory | GPU Usage | Complexity |
|----------|---------|--------|-----------|------------|
| Periodic batch (current) | Video duration | High | High | Low |
| Chunked processing (new) | 30-60s chunks | Medium | Medium | Medium |
| Frame-by-frame streaming | ~100ms | Low | Continuous | High |
| Event-triggered (motion) | ~1s | Low | On-demand | Medium |
| Adaptive chunking (scene-based) | Scene duration | Medium | Medium | High |

### Recommendation: Chunked Processing Mode

```python
class StreamingPipeline:
    """Process video in overlapping chunks for near-real-time analysis."""
    
    CHUNK_DURATION = 30  # seconds
    CHUNK_OVERLAP = 5    # seconds (for scene boundary detection)
    
    async def process_chunk(self, chunk_path: Path, chunk_index: int) -> ChunkResult:
        """Process a 30-second chunk independently."""
        scenes = await self.detect_scenes(chunk_path)
        frames = await self.extract_frames(chunk_path, scenes)
        analysis = await self.analyze_frames(frames)
        return ChunkResult(chunk_index, scenes, analysis)
    
    def merge_chunks(self, results: List[ChunkResult]) -> VideoIndex:
        """Merge overlapping chunks, deduplicate scenes at boundaries."""
        # Dedup scenes within CHUNK_OVERLAP seconds of chunk boundaries
        return self._merge_overlapping_scenes(results, overlap=CHUNK_OVERLAP)
```

### Config Fields

```python

streaming_chunk_overlap: int = 5
```

## 4. Video Quality Pre-Screening

### Problem Statement

The pipeline eagerly processes every frame/video regardless of quality. Blurry frames waste GPU cycles on YOLO/OCR/CLIP. Corrupted videos cause pipeline crashes. Static/locked frames (black screen, frozen) produce useless results.

### Research: Quality Assessment Methods

| Approach | VRAM | Speed | Detects | Implementation |
|----------|------|-------|---------|---------------|
| Laplacian variance | 0 | <1ms/frame | Blur | OpenCV built-in |
| BRISQUE (IQA) | ~50 MB | ~10ms/frame | Perceptual quality | pip install piq |
| CLAHE histogram | 0 | <1ms/frame | Over/underexposure | OpenCV |
| SSIM consecutive frames | 0 | ~2ms/frame | Static/frozen frames | OpenCV |
| FFmpeg error check | 0 | ~100ms/video | Corrupted files | FFmpeg process |

### Recommendation: Fast Pre-Screening Stage

```python
class VideoQualityScreener:
    """Fast pre-screening of video quality before GPU pipeline stages."""
    
    MIN_BLUR_THRESHOLD = 100   # Laplacian variance (lower = more blurry)
    MIN_BRIGHTNESS = 10        # Mean pixel value (lower = too dark)
    MAX_BRIGHTNESS = 245       # Mean pixel value (higher = overexposed)
    MAX_STATIC_FRAMES = 0.3   # Max 30% static frames allowed
    
    def screen_video(self, video_path: str) -> QualityReport:
        """Return quality report with pass/fail per metric."""
        frames = self._sample_frames(video_path, n=20)
        blur_scores = [self._laplacian_variance(f) for f in frames]
        brightness = [f.mean() for f in frames]
        static_pct = self._detect_static_frames(video_path)
        corrupted = self._check_corrupted(video_path)
        
        return QualityReport(
            is_blurry=any(s < self.MIN_BLUR_THRESHOLD for s in blur_scores),
            is_dark=np.mean(brightness) < self.MIN_BRIGHTNESS,
            is_overexposed=np.mean(brightness) > self.MAX_BRIGHTNESS,
            is_static=static_pct > self.MAX_STATIC_FRAMES,
            is_corrupted=corrupted,
            blur_score=np.mean(blur_scores),
            recommended_skip_stages=self._suggest_skip(is_blurry, is_dark),
        )
    
    def _laplacian_variance(self, frame: np.ndarray) -> float:
        """Classic blur metric — lower = blurrier."""
        gray = cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY)
        return cv2.Laplacian(gray, cv2.CV_64F).var()
```

### Config Fields

```python
quality_screening_enabled: bool = True
quality_min_blur_threshold: int = 100
quality_max_static_ratio: float = 0.3
quality_skip_ocr_on_blurry: bool = True
quality_skip_yolo_on_dark: bool = True
```

### Integration Point

New Stage 1.5 in the pipeline (between frame extraction and heavy GPU stages). When quality screening detects blurry/dark frames, OCR and YOLO are automatically skipped — saving VRAM and time.

**Estimated effort:** 1 day
**Impact:** 10-30% faster pipeline on low-quality videos, avoids wasted GPU cycles

## 5. Benchmarking Methodology & Performance Tracking

### Problem Statement

There is no systematic way to measure pipeline performance across versions. The README shows "~3-4 min for a 10-min video" but no per-stage breakdown, no VRAM tracking, no comparison across versions.

### Research: Benchmarking Infrastructure

```python
class PipelineBenchmark:
    """Systematic benchmarking of pipeline performance."""
    
    def __init__(self):
        self.metrics = defaultdict(list)
    
    @contextmanager
    def track_stage(self, stage_name: str):
        """Decorator to track per-stage metrics."""
        start_vram = self._get_vram_used()
        start_time = time.perf_counter()
        yield
        elapsed = time.perf_counter() - start_time
        peak_vram = self._get_vram_used() - start_vram
        self.metrics[stage_name].append({
            "wall_time_s": elapsed,
            "vram_mb": peak_vram,
            "timestamp": time.time(),
        })
    
    def _get_vram_used(self) -> int:
        """NVIDIA VRAM usage via pynvml."""
        try:
            import pynvml
            pynvml.nvmlInit()
            handle = pynvml.nvmlDeviceGetHandleByIndex(0)
            info = pynvml.nvmlDeviceGetMemoryInfo(handle)
            return info.used // (1024 * 1024)  # MB
        except Exception:
            return 0
    
    def report(self) -> str:
        """Generate markdown benchmark report."""
        total_time = sum(m[-1]["wall_time_s"] for m in self.metrics.values())
        rows = []
        for stage, measurements in sorted(self.metrics.items()):
            m = measurements[-1]
            pct = m["wall_time_s"] / total_time * 100
            rows.append(f"| {stage} | {m['wall_time_s']:.1f}s | {pct:.0f}% | {m['vram_mb']} MB |")
        return "\n".join(rows)
```

### Benchmarking CLI

```bash

# Compare across versions
python -m video_analysis benchmark --compare v0.19.0 v0.20.0

# Continuous tracking
BENCHMARK_TRACKING=true python -m video_analysis --video test.mp4
# Appends to data/benchmarks/history.json
```

### Key Metrics

| Metric | Tool | Granularity |
|--------|------|-------------|
| Wall-clock time per stage | Python timer | Per-video |
| Peak VRAM per stage | pynvml / nvidia-smi | Per-stage |

```python
# Pipeline logging
class PipelineLogger:
```

## 7. Federated Video Search

### Problem Statement

If a user has multiple video-analysis instances (e.g., home server + laptop + work server), there's no way to search across all of them. Each instance's ChromaDB is isolated.

### Research: Federated Search Approaches

| Approach | Complexity | Consistency | Latency | Implementation |
|----------|------------|-------------|---------|---------------|
| Export/import ChromaDB | Low | Stale | N/A | Manual, zip + scp |
| Multi-instance query | Medium | Near-real-time | Network | REST API fan-out |
| Shared ChromaDB server | Medium | Real-time | Network | ChromaDB HTTP mode |
| MCP tool federation | High | Varies | Varies | MCP server on each instance |

### Recommendation: MCP-Based Federation

```python
# Each instance exposes an MCP server. A "federation" tool queries all known instances.

FEDERATION_CONFIG = {
    "instances": [
        {"name": "home-server", "url": "http://192.168.1.100:7860/api/search"},
        {"name": "laptop", "url": "http://192.168.1.50:7860/api/search"},
    ]
}

async def federated_search(query: str, k: int = 5) -> List[dict]:
    """Query all instances and merge results."""
    results = []
    async with aiohttp.ClientSession() as session:
        for instance in FEDERATION_CONFIG["instances"]:
            async with session.post(
                instance["url"],
                json={"query": query, "k": k}
            ) as resp:
                data = await resp.json()
                for r in data["results"]:
                    r["_instance"] = instance["name"]
                    results.append(r)
    
    # Cross-instance de-duplication and re-ranking
    results.sort(key=lambda x: x["score"], reverse=True)
    return results[:k]
```

### Config Fields

```python

```

**Estimated effort:** 2-3 days (post-MCP-server)
**Impact:** Unified search across hardware/locations

## 8. Audio-Only Processing Mode

### Problem Statement

For audio-heavy content (podcasts, interviews, lectures), the visual pipeline stages (YOLO, OCR, CLIP) waste GPU cycles. User may want audio-only mode for faster processing.

### Config Fields

```python
processing_mode: str = "auto"   # auto, video_full, video_light, audio_only
```

### Behavior by Mode

| Mode | Audio Stages | Visual Stages | Use Case |
|------|-------------|---------------|----------|
| `video_full` | All | All | Movies, sports, vlogs |
| `video_light` | All | Scenes + CLIP only | Interviews, lectures |
| `audio_only` | Transcription + diarization | None | Podcasts, audiobooks |
| `auto` (default) | All | Quality-determined | Based on video_type heuristic |

### Implementation

Add `processing_mode` to config and filter stages in thepipeline:

```python
def _get_active_stages(self) -> List[str]:
    mode = self.config.processing_mode
    stages = AUDIO_STAGES  # Always include audio
    
    if mode == "video_full":
        stages.extend(VISUAL_STAGES)
    elif mode == "video_light":
        stages.extend(["detect_scenes", "classify_scenes", "generate_sprite"])
    elif mode == "audio_only":
        pass  # No visual stages
    elif mode == "auto":
        video_type = self._classify_video_type()
        stages.extend(self._get_stages_for_type(video_type))
    
    return stages
```

**Estimated effort:** 1 day
**Impact:** 50-75% faster processing for audio-heavy content
