## Major Finding: Qwen3-VL-30B-A3B — The New Optimal MLLM Backend

### Discovery

**Qwen3-VL-30B-A3B** (Alibaba, Apache 2.0) is a Mixture-of-Experts model with 30B total / 3B activated parameters. The `Instruct` and `Thinking` variants include:
- **256K native context** (expandable to 1M)
- **Advanced video understanding** — frame-by-frame description, second-level indexing, hours-long video processing
- **Visual agent abilities** — UI interaction, tool invocation
- **Enhanced multilingual OCR** — 32 languages
- **FP8 quantized versions** for efficient inference

### Why This Changes Everything for video-analysis

| Metric | VideoChat-Flash 2B (current) | Qwen3-VL-30B-A3B (proposed) | Improvement |
|--------|------------------------------|------------------------------|-------------|
| Parameters | 2B dense | 30B MoE (3B active) | ~50% more effective capacity |
| License | MIT | Apache 2.0 | More permissive |
| Context | ~128K tokens | 256K native (1M extended) | 2-8× more |
| Video understanding | MVBench 70.0 | Rivals Gemini-2.5-Pro | Much stronger |
| VRAM (FP8) | ~2.7 GB BF16 | ~6 GB (FP8 estimate) | Fits 12 GB budget |
| Agentic capabilities | Limited | Full visual tool use | Pipeline orchestration |
| Multimodal OCR | No | Yes (32 languages) | Reduces PaddleOCR dependency |

### Integration Plan

**Phase A — Evaluate**: Add Qwen3-VL as a third `video_mllm_backend` option alongside `"auto"`, `"videochat_flash"`, `"smolvlm2"`. Add `"qwen3_vl"` detection.

**Phase B — Test FP8 viability**: On RTX 4070, test `Qwen3-VL-30B-A3B-Instruct-FP8` for VRAM usage and latency.

**Phase C — Make default**: If FP8 fits, set `"auto"` backend to prefer Qwen3-VL over SmolVLM2.

### Model Size Options for 12 GB RTX 4070

| Model | Size | VRAM (BF16) | VRAM (FP8/INT8) | Fit? |
|-------|------|-------------|-----------------|------|
| Qwen3-VL-30B-A3B | 3B active | ~6 GB | ~3 GB (FP8) | ✅ |
| Gemma 3 12B | 12B | ~24 GB | ~12 GB (INT4/GGUF) | ⚠️ Tight |
| Gemma-3n-E2B-IT | ~5B eff. | ~10 GB | ~5 GB (FP8) | ✅ |
| SmolVLM2 2.2B | 2.2B | ~5.2 GB | N/A | ✅ (current) |
| GLM-4.6V-Flash 9B | 9B | ~18 GB | ~9 GB (FP8) | ⚠️ Tight |
| Qwen3.5-0.8B | 0.8B | ~1.6 GB | N/A | ✅ (for lightweight tasks) |

**Recommendation**: Add both Qwen3-VL-30B-A3B (primary, FP8) and Qwen3.5-0.8B (lightweight video classifier for pipeline orchestration). Keep SmolVLM2 and VideoChat-Flash as fallbacks.

## Finding 2: PaddleOCR v5 — Major Upgrade Available

### Current State
PaddleOCR v3/v4 is already in use. The project enables OCR via `OCR_ENABLED=true`.

### v5 Upgrade
- **PP-OCRv5**: 34.5M params, +5.1% recognition over v4_server, +13% end-to-end accuracy
- **109 languages** vs ~50 previously (adds Arabic, Devanagari, Tamil, Telugu, Cyrillic, etc.)
- **PP-StructureV3**: Hierarchical document parsing (tables, lists, headings)
- **PP-ChatOCRv4**: Key information extraction powered by LLM integration
- **Single model** covers 50 core languages efficiently
- Published as PaddleOCR 3.0 technical report (arXiv:2507.05595)

### Impact
- Better multilingual scene text reading in videos
- Improved accuracy on challenging frames (low-light, blurry, tilted text)
- Structured document output could enhance OCR scene descriptions in the pipeline

## Finding 3: Dependency Modernization Required

### Current vs. Actual Versions

| Package | pyproject.toml Bound | Installed | Issue |
|---------|---------------------|-----------|-------|
| `torch` | `>=2.1.0` | **2.12.1** | 4+ major versions behind; minimum too loose |
| `transformers` | `>=4.45.2` | **5.12.1** | v5 has breaking API changes over v4 |
| `sentence-transformers` | `>=2.5.0` | **5.6.0** | v3+ has breaking API changes |
| `fastapi` | `>=0.110.0` | **0.138.1** | Fine but should be bumped |
| `pillow` | `>=10.0.0` | **12.2.0** | Major updates |
| `opencv-python` | `>=4.9.0` | **4.13.0.92** | Updates with new algorithms |
| `chromadb` | `>=1.5.0` | **1.5.9** | Good — current |

### Actions Needed
1. **Update minimum bounds**: `torch>=2.5.0`, `transformers>=4.50.0`, `sentence-transformers>=2.7.0`
2. **Add missing deps**: `yt-dlp>=2024.12.0` to pyproject.toml dependencies (currently only in requirements.txt scripts)
3. **Pin major versions**: Add `>=5.0.0` for transformers, `>=2.7.0` for sentence-transformers
4. **Add optional deps**: `insightface>=0.7.0`, `ultralytics>=8.0.0`
5. **Consider uv**: The project already has `uv.lock` from early references; adopting `uv` for pip installs would speed up CI

## Finding 4: New Small Models for Pipeline Orchestration

### Qwen3.5-0.8B (Apache 2.0)
- **800M parameters** — smallest multimodal model with text+image+video support
- **262K context** — enough for long video descriptions
- **Multimodal at sub-1B** — can classify video type (lecture/sports/interview/etc.) in <200ms on GPU
- **Perfect for PipelineOrchestrator** (v0.17 research item #1)
- **200+ languages** support
- **VRAM**: ~1.6 GB — can run alongside most pipeline stages without unloading

### Gemma-3n-E2B-IT (Google)
- ~5B effective parameters (selective activation)
- Multimodal: text + image + audio + video
- Mobile-first architecture
- 32K token context shared across modalities
- **VRAM**: ~5 GB (FP8) — fits alongside other models
- Good for: lightweight video captioning, transcription, translation

### Integration
Both models should be:
1. Pipeline video classifier (replace OpenCLIP for scene type detection)
2. Optional lightweight MLLM for fast "what is this video?" queries
3. Part of the dynamic stage selector in PipelineOrchestrator

## Finding 5: ChromaDB vs. LanceDB — Stay or Migrate?

### Current Position
ChromaDB 1.5.9 is working well. 108/108 tests pass. No performance issues reported.

### Research Verdict
Based on the comprehensive vector DB comparison (4xxi, March 2026):
- **ChromaDB**: Best for 90% of projects. Simple, lightweight, production-ready. Handles millions of embeddings on 4-8 GB RAM.
- **LanceDB**: Interesting newcomer. Columnar format, handles larger-than-memory datasets well. Closest API match to ChromaDB.
- **Qdrant**: Better for complex metadata filtering (legal/financial apps) but adds operational complexity.
- **pgvector**: Practical if project already uses PostgreSQL, but video-analysis doesn't.

### Recommendation
**Stay on ChromaDB.** The existing codebase is tightly coupled to its API. Only consider LanceDB if:
- Vector count exceeds 5M (unlikely for personal video library)
- Performance becomes a bottleneck with >100 videos and >10K scenes
- Requirement for zero-copy reads or S3-native storage emerges

The **biggest change** from v0.17 research is that Qwen3-VL-30B-A3B (3B active, FP8) displaces the older VideoChat-Flash 2B as the primary MLLM recommendation. This has cascading effects:

1. **Retire VideoChat-Flash** as default — make it a legacy fallback

3. **Video MLLM** uses Qwen3-VL-30B-A3B FP8 for scene description + Q&A
4. **All transformers-dependent code** must be audited for v5 API compatibility
