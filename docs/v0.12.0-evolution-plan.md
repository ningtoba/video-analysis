## Finding 1: BGE-VL — The Game-Changer for Multimodal Embedding

### Discovery

**BAAI/bge-vl-base** (150M params, 299 MB, MIT license) is the single most impactful upgrade available. Key facts verified from HuggingFace and FlagEmbedding repo:

| Property | Value |
|----------|-------|
| **Model** | `BAAI/bge-vl-base` |
| **Params** | 149.6M (BF16) |
| **Size** | 299 MB |
| **VRAM** | ~0.8 GB |
| **Base** | OpenAI CLIP ViT-B/16 |
| **License** | MIT |
| **Downloads** | 75,277 all-time (stable, well-used) |
| **Last updated** | April 10, 2026 |
| **HF tag** | `sentence-similarity`, `multimodal-retrieval`, `embedding-model` |
| **Release** | December 2024 (base/large), March 2025 (MLLM) |

### Why This Changes Everything

Currently, the platform uses a **dual-model approach**:
- **Nomic Embed v1.5** (130M params, text-only) — default embedding
- **Qwen3-VL-Embedding-2B** (2B params, ~6GB VRAM) — optional multimodal

This means:
- Two models loaded = more GPU memory pressure
- Query-time routing between models is complex
- Multimodal (image+text) embedding requires the heavy Qwen3-VL
- Most users leave `MULTIMODAL_EMBEDDING=false` because of VRAM cost

With **BGE-VL-base**:
- **One model does everything**: text-only, image-text, composed retrieval
- **~0.8 GB VRAM** — barely more than Nomic Embed v1.5 (~0.5 GB)
- **MIT license** — same as Nomic Embed
- **Apache 2.0 / transformers-native** — uses trusted code pattern
- **sentence-transformers compatible** — `trust_remote_code=True`

### Integration Path

```python
# BGE-VL usage pattern (from FlagEmbedding docs):
from transformers import AutoModel
model = AutoModel.from_pretrained(
    "BAAI/BGE-VL-base",
    trust_remote_code=True,
)
model.set_processor("BAAI/BGE-VL-base")
model.eval()

with torch.no_grad():
    # Text-only embedding
    text_emb = model.encode(text="a person walking")
    # Image embedding
    img_emb = model.encode(images=[frame_path])
    # Composed (image + text)
    query_emb = model.encode(
        images=[frame_path],
        text="find similar scenes"
    )
    # Batch image embedding
    batch_emb = model.encode(images=[f1, f2, f3])
```

### Recommendation

**Replace SentenceTransformer + Qwen3-VL with BGE-VL-base as the single default embedding model.**

This simplifies the codebase dramatically:
- Remove the `_get_embedding()` vs `_get_multimodal_embedding()` bifurcation
- One unified embedding pipeline for text queries, image queries, and combined
- Frame images indexed alongside text in ChromaDB directly
- Cross-video search benefits from true visual matching (not just text descriptions)
- Cleaner config: remove `multimodal_embedding_enabled`, `multimodal_embedding_model`

> **⚠️ Caveat**: BGE-VL requires `transformers>=4.45.2` (current pin is >=4.40.0). Need to verify compatibility and bump the pin.

### BGE-VL-large Alternative

If VRAM allows (~1.5 GB):
- BAAI/bge-vl-large (428M params, 855 MB)
- Better accuracy for complex multimodal queries
- Still dramatically lighter than Qwen3-VL-Embedding-2B (~6 GB)

## Finding 2: TV-RAG — Temporal-Aware Video Retrieval

### Discovery

**TV-RAG** (ACM Multimedia 2025, arXiv:2512.23483, December 2025) introduces two mechanisms directly applicable to this platform:

#### 2a. Time-Decay Retrieval

Instead of pure cosine similarity, weight retrieval scores by temporal proximity to the query's implied temporal context:

```python
def temporal_score(query_emb, chunk_emb, query_time, chunk_time, decay_rate=0.1):
    similarity = cosine_similarity(query_emb, chunk_emb)
    time_distance = abs(query_time - chunk_time) if query_time else 0
    temporal_weight = exp(-decay_rate * time_distance)
    return similarity * temporal_weight
```

**Current platform state**: `expand_temporal_context()` adds ±1 neighbor scenes after retrieval. TV-RAG shows we should weight retrieved chunks by temporal proximity during retrieval itself.

#### 2b. Entropy-Weighted Key-Frame Sampling

Instead of uniform 1-frame-per-2-seconds sampling, select frames with the highest information gain (entropy) while maintaining uniform temporal coverage:

- Compute pixel-level or feature-space entropy for each candidate frame
- Select top-N frames per time window that maximize coverage
- Reduces redundant frames from static scenes by 30-50%

**Current platform state**: Uniform sampling at 0.5 fps + optional CLIP dedup. Entropy-weighted sampling is complementary — it selects *informative* frames before dedup removes *similar* ones.

### Implementation Plan

1. **Temporal weighting in `rag.py`**: Add `temporal_decay_rate` config field. When > 0, apply `exp(-decay_rate * time_distance)` to retrieval scores.
2. **Entropy sampler in `pipeline.py`**: Add `_sample_keyframes_entropy()` method that ranks frames by entropy and selects top-N per scene.
3. **Query time extraction**: When a user query references a timestamp (e.g., "at 5:30"), parse it and use as `query_time` for temporal weighting.

## Finding 3: Multi-Granularity Chunking

### Current Approach

Three chunk types in `index_video()`:
1. Scene chunks (variable length — transcript + frame descriptions + objects)
2. Frame chunks (individual frame context)
3. Transcript chunks (500-char sliding window, 50-char overlap)

### Issues Identified

- Frame chunks are often too short (< 20 chars) for semantic matching
- Scene chunks can be very long (multi-minute scenes with dense dialog)
- No fixed-window chunks for cross-scene temporal queries
- No sliding window across scene boundaries

### Recommended: Quad-Chunk Strategy

| Chunk Type | Size | Content | Use Case |
|-----------|------|---------|----------|
| **Scene** | Variable | Transcript + descriptions + objects + OCR + actions | Initial retrieval, temporal context |
| **Fixed-window (60s)** | ~60 seconds | Transcript segments + summary descriptions | Cross-scene queries, long events |
| **Sliding (15s overlap)** | ~30 seconds | Transcript only, partial overlap | Fine-grained temporal localization |
| **Frame** | Per-frame | Description + objects + OCR + action | Direct frame-level retrieval |

```python
CHUNK_SPEC = [
    ("scene", None, {"include": ["transcript", "descriptions", "objects", "ocr", "actions"]}),
    ("fixed_60s", 60.0, {"overlap": 15.0, "include": ["transcript"]}),
    ("sliding_30s", 30.0, {"overlap": 15.0, "include": ["transcript"]}),
    ("frame", None, {"include": ["description", "objects", "ocr", "action"]}),
]
```

Each chunk type gets a `chunk_type` metadata field in ChromaDB so queries can target specific granularities.

## Finding 4: GPU Memory Management Audit

### Current State

Pipeline stages in order with model memory impact:

| Step | Stage | Model | VRAM | Cleanup? |
|------|-------|-------|------|----------|
| 1-4 | FFmpeg + frame extraction | None | 0 | N/A |
| 5 | Transcribe | faster-whisper large-v3 | ~4 GB | ❓ Loaded once, unloaded? |
| 6 | Diarization | PyAnnote (CPU) | 0 | N/A |
| 7 | Object detection | YOLO | ~1 GB | ❓ Stays in GPU |
| 8 | OCR | PaddleOCR (CPU) | 0 | N/A |
| 9 | Scene classification | OpenCLIP | ~2 GB | ❓ Stays in GPU |
| 10 | Action recognition | X-CLIP | ~4 GB | ❓ Stays in GPU |
| 11 | Transcript assignment | None | 0 | N/A |
| 12 | Sprite sheet | FFmpeg | 0 | N/A |
| 13 | Index build | None | 0 | N/A |

**Problem**: If all GPU models are loaded simultaneously, peak VRAM usage = ~4+1+2+4 = **~11 GB** — dangerously close to RTX 4070's 12 GB limit.

### Recommended: Systematic Model Unloading

```python
import gc
import torch

def _unload_model(self, model_attr: str):
    """Safely unload a model from GPU memory."""
    model = getattr(self, model_attr, None)
    if model is not None:
        setattr(self, model_attr, None)
        del model
    gc.collect()
    torch.cuda.empty_cache()
    torch.cuda.synchronize()

# Usage after each GPU stage:
# After transcription completes
self._unload_model("_whisper_model")
# After object detection
self._unload_model("_yolo_model")
# After scene classification
self._unload_model("_clip_model")
# After action recognition
self._unload_model("_action_recognizer")
```

## Finding 5: Graceful Shutdown (SIGTERM/SIGINT)

### Current State

**No signal handling.** If `SIGTERM` or `SIGINT` arrives mid-processing:
- ChromaDB could be left in an inconsistent state (partial index write)
- Temporary files (audio, frames) are not cleaned up
- GPU models remain loaded and consume VRAM

### Implementation

```python
import signal

class VideoPipeline:
    def __init__(self, config=None):
        self._shutdown_requested = False
        signal.signal(signal.SIGTERM, self._handle_signal)
        signal.signal(signal.SIGINT, self._handle_signal)

    def _handle_signal(self, signum, frame):
        logger.warning(f"Signal {signum} received — finishing current operation...")
        self._shutdown_requested = True

    def process(self, video_path) -> Optional[VideoIndex]:
        # ... existing process ...
        for scene in scenes:
            if self._shutdown_requested:
                logger.info("Shutdown requested — saving partial results")
                index = self._build_partial_index(video_path, scenes, ...)
                # Save what we have
                return index
        # ... normal completion ...
```

Also add to `__main__.py` for the CLI/UI process:
```python
def main():
    shutdown_event = threading.Event()
    def handle_sigterm(signum, frame):
        logger.info("Shutting down gracefully...")
        shutdown_event.set()
    signal.signal(signal.SIGTERM, handle_sigterm)
    signal.signal(signal.SIGINT, handle_sigterm)
    # Launch with shutdown_event in loop
```

## Finding 6: Embedding Prefix Normalization

### Current State

`_get_embedding()` in `rag.py` sends raw query/document text without prefixes:

```python
emb = self._embedding_model.encode(text, normalize_embeddings=True)
```

### The Problem

Modern embedding models (Nomic Embed, BGE, E5, Jina) are trained with different prefixes for queries vs. documents. Omitting prefixes loses 5-10% retrieval accuracy.

### Recommendation

```python
EMBEDDING_PREFIXES = {
    "nomic-ai/nomic-embed-text-v1.5": {
        "query": "search_query: ",
        "document": "search_document: ",
    },
    "BAAI/bge-small-en-v1.5": {
        "query": "Represent this query for searching: ",
        "document": "Represent this document for retrieval: ",
    },
    # BGE-VL doesn't need explicit prefixes — handled internally
}
```

When `multimodal_embedding_enabled=False` (Nomic path), apply prefixes. When BGE-VL is the backend, its `model.encode()` handles internally.

```yaml
dcgm-exporter:
  image: nvcr.io/nvidia/k8s/dcgm-exporter:3.3.8-3.6.0-ubuntu22.04

  cap_add:
    - SYS_ADMIN

    - "9400:9400"
```

### 7b. Caddy Reverse Proxy

For production deployment behind a domain (as discussed in v0.11.0 research):

```yaml
# docker-compose.prod.yml additions
caddy:
  image: caddy:2-alpine
  ports:
    - "80:80"
    - "443:443"
  volumes:
    - ./Caddyfile:/etc/caddy/Caddyfile
```

With a `Caddyfile` that handles Gradio WebSocket connections:
